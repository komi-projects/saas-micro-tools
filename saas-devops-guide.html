DevOps for SaaS

DevOps isn't just about tools — it's about culture. The goal: ship code to production safely, quickly, and repeatedly. Small changes, frequent deploys, fast recovery. The opposite: big-bang releases, manual processes, weekend deploys, "don't touch it, it works."

For SaaS, DevOps is critical. Your product is live 24/7. Every minute of downtime is lost revenue and trust. Every slow deploy is delayed value.

Core Practices

🔄 CI/CD (Continuous Integration / Continuous Deployment)
Every code change is built, tested, and deployed automatically. Git push → tests run → deploy to staging → manual approval → deploy to production. Tools: GitHub Actions, GitLab CI, CircleCI, Jenkins. Target: deploy to production > 10x per week.
📦 Infrastructure as Code (IaC)
Define infrastructure in code (Terraform, CloudFormation, Pulumi), not manual clicks. Version-controlled, reviewable, reproducible. Spin up a new environment in minutes. Disaster recovery: run the script, rebuild everything.
🐳 Containers & Orchestration
Docker containers package your app + dependencies. Kubernetes (or managed: ECS, GKE, AKS) orchestrates scaling, health checks, rollouts. "It works on my machine" becomes "It works in a container."
🧪 Automated Testing
Unit tests (fast, isolated), integration tests (APIs, DBs), and end-to-end tests (full user flows). Run on every PR. Block deploy if tests fail. Test coverage > 70% is a good target. 100% coverage with bad tests is worse than 50% with good tests.
📊 Monitoring & Alerting
If you can't measure it, you can't improve it. Metrics (CPU, memory, request latency), logs (structured, searchable), traces (request flows across services). Alert on symptoms (error rate > 1%), not causes (disk > 90%). Page on revenue-impacting issues only.
🔄 Feature Flags
Deploy code, release features separately. Merge incomplete features behind a flag. Enable for 5% of users, then 50%, then 100%. If something breaks, flip the flag off — no rollback needed. Tools: LaunchDarkly, Split, Unleash, or simple config flags.

Deployment Strategy

Blue-Green Deployment: Two identical environments. Deploy to "green," test, switch traffic from "blue." Instant rollback: switch back. Zero downtime. Requires double infrastructure.

Canary Deployment: Deploy to 5% of servers, monitor, then 25%, then 50%, then 100%. If error rate spikes, halt. Gradual, safe, data-driven.

Rolling Deployment: Replace servers one by one. No extra infrastructure needed. Slower, but cost-effective. Good for stateless apps.

Database Migrations: Never drop columns in the same deploy that stops using them. Three-step process: (1) Add new column, (2) Update code to write to both, (3) Remove old column. Backward-compatible = zero-downtime.

DevOps Tooling Stack

Code & Collaboration: GitHub, GitLab, Bitbucket
CI/CD: GitHub Actions, GitLab CI, CircleCI, Travis CI
IaC: Terraform, Pulumi, AWS CloudFormation
Containers: Docker, Kubernetes, ECS, GKE
Monitoring: Datadog, New Relic, Prometheus + Grafana
Logging: ELK Stack, Splunk, Papertrail, LogDNA
Feature Flags: LaunchDarkly, Split, Unleash
Secrets: Vault, AWS Secrets Manager, Doppler

Rule: Pick tools your team knows. A mediocre tool used well beats a perfect tool no one understands. Start simple, add complexity when needed.

DevOps Metrics (DORA)

The DevOps Research Alliance (DORA) identified 4 key metrics:

• Deployment Frequency: How often you deploy. Elite: on-demand (> 1/day). High: 1/week-1/day. Medium: 1/month-1/week. Low: < 1/month.
• Lead Time for Changes: Time from commit to production. Elite: < 1 hour. High: < 1 day. Medium: 1 day-1 week. Low: > 1 month.
• Change Failure Rate: % of deploys causing incidents. Elite: < 5%. High: 5-15%. Medium: 15-30%. Low: > 30%.
• Mean Time to Recovery (MTTR): Time to recover from failure. Elite: < 1 hour. High: < 1 day. Medium: 1 day-1 week. Low: > 1 week.

Elite performers: Deploy multiple times per day, recover in under an hour, and have < 5% failure rate. They're not more careful — they're better at detecting and recovering.

On-Call Best Practices

1. Rotate on-call: Weekly rotations. No one should be on-call > 25% of the time. Compensate: extra PTO or pay.

2. Runbooks for everything: "If alert X fires, check Y, run Z, escalate to W." No one should wake up at 3 AM and guess what to do.

3. Post-mortems without blame: What happened? How did we detect it? How did we fix it? How do we prevent it? Document and share. The goal is learning, not punishment.

4. Reduce alert noise: If an alert fires and no action is needed, it's not an alert — it's a metric. Page only on revenue-impacting issues. Everything else: Slack notification during business hours.
