🏳️ Feature Flags

Deploy safely and roll out gradually without risk.

Why Feature Flags Are Essential

Feature flags (also called feature toggles) are the most important technique for modern SaaS deployment. They decouple deployment from release, allowing you to ship code to production without exposing it to users. This means safer deployments, faster iteration, and the ability to test in production with real users.

Without feature flags, every deployment is a high-stakes event. With feature flags, you can deploy 50 times a day, enable features for 1% of users, watch metrics, and gradually roll out — or instantly disable if something breaks. It's the difference between "pray and deploy" and "deploy with confidence."

This guide covers how to implement, manage, and optimize feature flags in your SaaS product.

Feature Flag Types

1. Release Toggles
Hide incomplete features from users. • Long-lived flags for features in development • Code ships to production but stays hidden • Team can test internally without affecting users • Removed once feature is complete and stable • Example: "New dashboard" flag — dev team sees it, users don't
2. Experiment Toggles
A/B test features with user segments. • Random or targeted user assignment • Measure impact on key metrics • Statistical significance before declaring winner • Automatically roll out winner, disable loser • Example: "New checkout flow" — 50% see old, 50% see new
3. Ops Toggles
Operational controls for system health. • Circuit breakers: Disable failing features automatically • Rate limiting: Throttle features during high load • Maintenance mode: Show "under maintenance" instead of errors • Kill switches: Instantly disable problematic features • Example: "Third-party API integration" — disable if API is down
4. Permission Toggles
Feature access based on user attributes. • Plan-based: "Pro feature" only for Pro users • Role-based: Admin features only for admins • Beta access: New features for power users first • Geography: Features available only in certain regions • Example: "Advanced reporting" — only available on Business plan
5. Canary Toggles
Gradual rollout with real-time monitoring. • 1% of traffic → 5% → 25% → 100% • Automatic rollback if error rate spikes • Target by user segment, geography, or random • Monitor business metrics, not just technical ones • Example: "New search algorithm" — roll out gradually, watch conversion

Feature Flag Implementation

1. Code integration:

• SDK: Import the feature flag SDK into your application • Evaluation: Check flag state before executing feature code • Default behavior: If flag service is down, default to "off" (safe) • Performance: Evaluate flags locally (cached) to avoid latency • Context: Pass user attributes, device, location for targeting

2. Flag management:

• Dashboard: Web UI for creating, editing, and monitoring flags • Rules engine: IF user.plan == "pro" AND user.country == "US" THEN enable • Targeting: User IDs, segments, percentages, custom attributes • Scheduling: Enable flags at specific dates/times • Variations: Multiple variants for A/B testing (not just on/off) • Dependencies: Flag B only enabled if Flag A is on

3. Monitoring:

• Flag usage: How often is each flag evaluated? • Performance impact: Does the flag check add latency? • Error rates: Are flagged features causing more errors? • Business metrics: Impact on conversion, retention, revenue • Flag health: Stale flags, unused flags, conflicting flags • Audit log: Who changed what flag and when?

Feature Flag Best Practices

1. Naming conventions:

• Descriptive: "new-checkout-flow" not "flag-47" • Scoped: "payment.new-checkout-flow" for organization • Temporary vs. permanent: Prefix temporary flags ("temp-" or "exp-") • Owner: Include team or feature name • Date: Add creation date for stale flag detection • Example: "exp-2024-03-payment-new-checkout"

2. Lifecycle management:

• Create: Flag created when feature development starts • Develop: Team uses flag for internal testing • Test: Enable for beta users, monitor metrics • Rollout: Gradually increase percentage • General availability: 100% of users, feature is default • Deprecate: Remove flag, make feature permanent • Archive: Remove flag code, clean up references • Maximum flag age: 30 days for experiment flags, 90 days for release flags

3. Safety:

• Default off: New flags default to disabled (fail-safe) • Local evaluation: Cache flag state, don't call service on every request • Fallback: If flag service is unavailable, default to known-good state • Circuit breaker: If flag evaluation fails repeatedly, disable feature • Testing: Unit tests with flags on and off • Documentation: Every flag must have an owner, description, and removal date

Feature Flag Tools

Commercial:

• LaunchDarkly: Enterprise feature management, targeting, experimentation • Split: Feature flags + A/B testing, data-driven decisions • Optimizely: Full-stack experimentation platform • Flagsmith: Open-source core, commercial hosting • Harness: Feature flags + continuous delivery

Open-source:

• Unleash: Open-source feature flagging, self-hosted • GrowthBook: Open-source A/B testing + feature flags • Flipper: Ruby feature flags (used by GitHub) • Flipt: Self-hosted feature flag solution • PostHog: Open-source product analytics with feature flags

Build your own:

• Simple: Database table with flag names and boolean values • Advanced: Redis-backed with caching, targeting rules, and analytics • DIY trade-offs: Free, but requires maintenance, monitoring, and security

Feature Flag Mistakes

1. Flag sprawl: Hundreds of flags, nobody knows what they do. Implement flag lifecycle management and regular cleanup.

2. No removal process: Flags stay in code forever, creating technical debt. Every flag needs an expiration date and removal ticket.

3. Inconsistent evaluation: Frontend says flag is on, backend says off. Use consistent evaluation logic and shared context.

4. Performance impact: Calling flag service on every request adds latency. Cache flag state locally and evaluate asynchronously.

5. Security exposure: Flags control access to sensitive features. Ensure flag evaluation can't be bypassed by users.

6. No monitoring: You have 50 flags but no idea which are active, which are stale, or which are causing issues. Track flag usage and health.

7. Testing gaps: Testing only with flags on (or only off). Test both states, and test flag transitions (on → off → on).
