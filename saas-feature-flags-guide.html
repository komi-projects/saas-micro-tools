Ship safely. Test in production. Control everything.

What Are Feature Flags?

A feature flag (or feature toggle) is a software development practice that allows you to enable or disable features without deploying new code. It's an if-statement in your code controlled by a configuration system: if flag is ON, show the new feature; if OFF, show the old behavior.

Feature flags decouple deployment from release. You can deploy code to production and keep it hidden. Then, when you're ready, flip the flag to enable it for everyone — or for a subset of users. If something goes wrong, flip it back. No deploy, no rollback, no downtime.

Without feature flags: You merge code, deploy it, and hope it works. If it doesn't, you roll back the entire deploy, reverting potentially dozens of unrelated changes.

With feature flags: You merge code, deploy it dark, test it on 1% of users, gradually increase to 100%, and if anything breaks, disable just that feature in seconds.

Types of Feature Flags

1. Release Flags (Short-lived)
Hide incomplete features until they're ready. Used during development, removed after launch. • "New dashboard" — team works on it for 3 weeks, flag keeps it hidden • "API v2" — code is deployed but not routed to until tested • Remove after feature is stable (debt management)
2. Experiment Flags (A/B Testing)
Route users to different experiences and measure outcomes. • "Pricing page v2" — 50% see new design, measure conversion • "Onboarding flow B" — test against flow A • "ML model v2" — 10% get new model, measure accuracy • Run for days/weeks, then pick winner and clean up
3. Operational Flags (Kill Switches)
Disable features during incidents without deploying. • "Third-party API integration" — if API goes down, disable the feature • "New search algorithm" — if performance degrades, switch back to old • "Email sending" — if queue backs up, pause non-critical emails • Long-lived. Never remove. These are safety valves.
4. Permission Flags (Entitlement)
Control feature access by user, plan, or tenant. • "Advanced analytics" — only Pro plan users see it • "Beta features" — only users who opted in • "Enterprise SSO" — only enterprise tenants • Long-lived. Part of your product packaging.
5. Progressive Delivery
Gradually roll out to manage risk. • "1% → 5% → 25% → 50% → 100%" over a week • Geographic rollout: US first, then EU, then APAC • Time-based: enable during business hours, disable overnight initially • Automated: if error rate spikes, auto-rollback to 0%

Feature Flag Architecture

Simple (config file):

• JSON/YAML config file deployed with the app • Requires deploy to change flags • Fine for 2-3 flags in small teams • Doesn't support user targeting or percentages • Risk: changing a config file is almost as slow as changing code

Database-driven:

• Flags stored in database, read at runtime • No deploy needed to flip flags • Supports targeting (user ID, plan, region) • Add caching (Redis) to avoid DB hits on every request • Good for 10-50 flags

Feature flag service (LaunchDarkly, Split, Flagsmith):

• Managed service with SDKs for every language • Real-time updates (WebSocket/SSE) • Advanced targeting, A/B testing, analytics • Audit logs, approval workflows • Good for 50+ flags or teams that need advanced capabilities • Cost: $50-500/month depending on MAUs

Self-hosted (Unleash, Flipt, GrowthBook):

• Open-source, run on your infrastructure • Full control, no vendor lock-in • Requires setup and maintenance • Good for privacy-sensitive or cost-conscious teams

Implementing Feature Flags

Code example (pseudocode):

// Without feature flag
function getDashboard(user) {
  return oldDashboard(user);
}

// With feature flag
function getDashboard(user) {
  if (featureFlags.isEnabled('new-dashboard', user)) {
    return newDashboard(user);
  }
  return oldDashboard(user);
}

Best practices:

• Default to off: New flags should default to false. Explicit enable only. • Remove old code: After a flag has been 100% on for 2+ weeks, remove the old code path and the flag. Flag debt accumulates. • Test both paths: Unit tests should cover flag ON and flag OFF states. • Evaluate once per request: Don't check the flag service on every function call. Evaluate at request start and pass the result. • Don't nest flags: Flag A inside flag B inside flag C creates exponential complexity. Max 2 levels deep. • Document flags: Every flag needs an owner, a description, and a removal date. Use a flag registry. • Audit changes: Who changed what flag, when, and why. Critical for incident investigation.

Feature Flag Lifecycle

1. Creation: Developer adds flag during feature development. Default: OFF for everyone. Flag has owner, description, and target removal date (2-4 weeks post-launch).

2. Testing: Enable for internal team and beta users. Monitor error rates, performance, and user feedback. Fix issues without affecting all users.

3. Gradual rollout: Enable for 1% of users → 5% → 25% → 50% → 100%. Monitor metrics at each stage. Automated rollback if error rate or latency degrades.

4. Full rollout: 100% of users have the feature. Monitor for 1-2 weeks to ensure stability.

5. Cleanup: Remove the flag and the old code path. Flag registry updated. Code is now permanently on. This is the step most teams skip — and it creates technical debt.

6. Archive: Flag metadata kept for audit purposes but removed from active system.

Feature Flag Anti-Patterns

❌ Flag debt: Hundreds of flags that are always ON but never cleaned up. Each flag adds complexity, testing burden, and risk. Set a "flag retirement" sprint every quarter.

❌ Flags as configuration: Using flags for things that should be settings (timezone, currency, branding). Flags are for code paths, not data. Blurring the line creates confusion.

❌ Too many flags: A feature split into 20 micro-flags. "Show button A" + "Show button B" + "Enable animation" + "Change color"... Combine related changes into one flag.

❌ Flags without monitoring: Rolling out a feature without tracking error rates, performance, or business metrics. You're flying blind.

❌ No rollback plan: Enabling a flag with no way to quickly disable it. The whole point of flags is fast rollback. If disabling takes > 1 minute, fix your tooling.

❌ Flags in database migrations: Never use feature flags for schema changes. Migrations must be deterministic. Flags can be used for application logic only.

Feature Flag Tools

Managed services:

• LaunchDarkly: The market leader. Comprehensive, expensive, reliable. Best for enterprise. • Split: Strong analytics, data-driven decisions. Good for experimentation. • Optimizely: Full experimentation platform. Flags + A/B testing + personalization. • Flagsmith: Open-source core, managed option. Good middle ground. • ConfigCat: Simple, affordable, good for smaller teams. • PostHog: Open-source product analytics with feature flags. Great for startups.

Open source / Self-hosted:

• Unleash: Mature open-source feature toggle system. Enterprise support available. • Flipt: Modern, fast, gRPC-native. Good for high-throughput systems. • GrowthBook: Open-source A/B testing + feature flags. Great for experimentation. • Flagger (GitOps): Kubernetes-native progressive delivery. Automated canary analysis.

Build your own:

• For 5-10 flags, a database table + Redis cache is sufficient. • For more, consider open-source solutions before building custom. • Custom solutions tend to lack the UI, analytics, and safety features of mature products.

Feature Flag Checklist

☐ Every flag has an owner and removal date
☐ Flags are created in OFF state by default
☐ Both flag paths (ON/OFF) are tested in CI
☐ Rollout is gradual (1% → 100%) with monitoring
☐ Flags are removed 2-4 weeks after full rollout
☐ Flag changes are audited (who, when, why)
☐ On-call can disable any flag in < 1 minute
☐ Flags are not used for database schema changes
☐ No more than 50 active flags at any time
☐ Quarterly flag cleanup sprint scheduled
☐ Flag service has 99.9% uptime SLA
☐ Fallback behavior works if flag service is unavailable
