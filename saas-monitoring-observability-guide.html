📡 Monitoring & Observability

Know what's happening before customers complain.

Why Observability is Non-Negotiable

You can't improve what you don't measure. In SaaS, every minute of downtime costs money, every slow query costs customers, and every error costs trust. Observability — the combination of metrics, logs, and traces — gives you the visibility to detect problems before customers do, debug incidents in minutes instead of hours, and make data-driven decisions about where to invest engineering effort.

The best SaaS companies don't just monitor uptime; they monitor business metrics (signup rate, conversion, revenue), user experience (page load times, API latency, error rates), and system health (CPU, memory, disk, queue depth). They set alerts that wake people up for real problems and stay quiet for noise. They build dashboards that tell the story of their system's health at a glance.

This guide covers how to build observability into your SaaS from day one — the tools, the metrics, the alerts, and the culture of reliability.

The Three Pillars of Observability

1. Metrics (What's happening?)
Numerical data over time. • Infrastructure: CPU, memory, disk I/O, network throughput • Application: Request rate, error rate, latency (p50, p95, p99), queue depth • Business: Signups, conversions, revenue, churn, active users • Format: Time-series data, typically aggregated (average, sum, count) • Tools: Prometheus, Datadog, New Relic, CloudWatch, Grafana • Best practice: Use percentiles, not averages. p99 latency tells you the real user experience.
2. Logs (What happened?)
Detailed event records. • Structured logs: JSON format with consistent fields (timestamp, level, message, trace_id) • Correlation IDs: Same ID across all logs for a single request • Levels: DEBUG (development), INFO (normal), WARN (attention), ERROR (investigate), FATAL (wake up) • Retention: 7-30 days hot, 90 days-1 year cold, then archive • Tools: ELK stack, Splunk, Datadog, CloudWatch Logs, Loki • Best practice: Log context, not just messages. "Payment failed" is useless. "Payment failed for customer 123, amount $50, error: card_declined" is actionable.
3. Traces (Where did it happen?)
Request journey across services. • Distributed tracing: Follow a request through API → service → database → cache → third-party API • Spans: Each operation in the trace (HTTP request, DB query, function call) • Trace ID: Unique identifier propagated across all services • Visualization: Waterfall view showing timing and dependencies • Tools: Jaeger, Zipkin, Datadog APM, New Relic, Honeycomb • Best practice: Instrument every service, not just the entry point. A slow query deep in the stack affects the user experience too.

Key SaaS Metrics to Monitor

Golden Signals (from Google's SRE book):

• Latency: How long do requests take? Track p50, p95, p99. p99 is what your unhappiest users experience. • Traffic: How many requests per second? Sudden drops mean problems; sudden spikes mean attacks or virality. • Errors: What percentage of requests fail? 4xx = client errors (maybe okay), 5xx = server errors (always bad). • Saturation: How full are your resources? CPU >80%? Memory >90%? Queue depth growing? These predict outages.

Business Metrics:

• Signup rate: New signups per day/hour • Activation rate: % of signups who reach "aha moment" • Conversion rate: Free → paid percentage • Revenue: MRR, ARR, daily revenue • Churn: Monthly churn rate, churn by cohort • NPS: Net Promoter Score trends • Support tickets: Tickets per customer, resolution time

Security Metrics:

• Failed login attempts: Brute force detection • API abuse: Unusual request patterns • Data access anomalies: Unusual data export volumes • Vulnerability scan results: Critical/high/medium/low counts

Alerting Strategy

1. Alert severity:

• P0 (Critical): Site down, payments failing, data loss. Page immediately, any time. • P1 (High): Major feature broken, significant performance degradation. Page during business hours. • P2 (Medium): Minor issue, workaround exists. Ticket for next business day. • P3 (Low): Cosmetic issue, nice-to-have improvement. Backlog item.

2. Alert rules:

• Error rate >1% for 5 minutes → P1 alert • p99 latency >500ms for 10 minutes → P2 alert • CPU >80% for 15 minutes → P2 alert • Disk space >85% → P2 alert (P1 if >95%) • Failed payment rate >5% → P0 alert • Signup rate drops >50% vs. yesterday → P1 alert • 5xx errors >0.1% → P1 alert

3. Alert fatigue prevention:

• Require sustained violation (5+ minutes, not 1 spike) • Group related alerts (if 10 services alert, send 1 grouped notification) • Auto-resolve when metric recovers • Regular review: If an alert fires and nobody acts, change or remove it • Runbooks: Every alert should link to a runbook with debugging steps

Dashboard Design

1. Executive dashboard:

• Revenue (MRR, ARR, growth) • Customer metrics (total, active, new, churned) • System health (uptime %, incident count) • Update frequency: Daily or weekly (not real-time) • Audience: CEO, board, investors

2. Engineering dashboard:

• Error rates by service • Latency percentiles by endpoint • Infrastructure utilization • Deployment status and frequency • Update frequency: Real-time • Audience: Engineering team, on-call engineer

3. Product dashboard:

• Feature adoption (usage by feature) • Funnel analysis (signup → activation → paid) • A/B test results • User engagement (DAU/MAU, session duration) • Update frequency: Hourly or daily • Audience: Product managers, designers, marketers

Observability Tools

All-in-one platforms:

• Datadog: Market leader, comprehensive, expensive ($15/host/month + ingestion) • New Relic: Strong APM, good for application monitoring, pricing by data volume • Dynatrace: AI-powered observability, enterprise-focused • Splunk: Best-in-class log analysis, expensive for high volumes

Open-source stack:

• Prometheus + Grafana: Metrics collection and visualization (free, self-hosted) • ELK stack (Elasticsearch, Logstash, Kibana): Log aggregation and search • Jaeger or Zipkin: Distributed tracing • Loki: Lightweight log aggregation (pairs well with Grafana) • VictoriaMetrics: Prometheus-compatible, better performance at scale

Cloud-native:

• AWS CloudWatch: Native AWS integration, basic but functional • Google Cloud Monitoring: Strong for GCP workloads • Azure Monitor: Best for Microsoft stacks • Vercel/Netlify Analytics: Frontend performance for Jamstack apps

Monitoring Checklist

☐ Metrics collection configured (infrastructure, application, business) ☐ Golden signals monitored (latency, traffic, errors, saturation) ☐ Business metrics dashboards created (revenue, signups, churn) ☐ Log aggregation set up (structured logs, correlation IDs) ☐ Log retention policy defined (7-30 days hot, 90+ days archive) ☐ Distributed tracing implemented (trace IDs across services) ☐ Alert severity levels defined (P0, P1, P2, P3) ☐ Alert rules configured with sustained thresholds ☐ On-call rotation established (who gets paged, escalation path) ☐ Runbooks created for common alerts (debugging steps) ☐ Executive dashboard created (revenue, customers, uptime) ☐ Engineering dashboard created (errors, latency, deployments) ☐ Product dashboard created (adoption, funnels, engagement) ☐ Incident response process defined (detect, respond, resolve, postmortem) ☐ Postmortem template created (timeline, root cause, action items) ☐ SLOs/SLIs defined (what does "good" mean for each service?) ☐ Error budget policy established (when to halt feature work for reliability) ☐ Synthetic monitoring set up (ping tests, user journey simulation) ☐ RUM (Real User Monitoring) configured (frontend performance from real users) ☐ Anomaly detection enabled (unusual patterns without manual thresholds) ☐ Cost monitoring configured (infrastructure spend by service) ☐ Health checks for all services (endpoint that returns 200 if healthy) ☐ Status page created (public page with incident history) ☐ Team trained on observability tools and incident response
