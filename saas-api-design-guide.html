🔌 API Design

Build APIs developers love. Drive adoption. Enable integrations.

Why API Design Matters

Your API is your product for developers. A well-designed API reduces integration time from weeks to hours. A poorly designed API creates support tickets, frustrated customers, and churn.

Great APIs are:

• Predictable: Consistent patterns developers can learn once and apply everywhere.
• Discoverable: Self-documenting with clear errors and examples.
• Reliable: Stable versions, clear deprecation policies, high uptime.
• Fast: Response times < 200ms. Pagination for large datasets.
• Secure: OAuth 2.0, rate limiting, scoped permissions.

REST API Best Practices

📋 Resource-Based URLs
Use nouns, not verbs. /customers not /getCustomers. /invoices/123 not /getInvoice?id=123. HTTP methods (GET, POST, PUT, DELETE) are the verbs.
🔢 Consistent Naming
Plural nouns: /customers, /orders, /products. Kebab-case: /billing-addresses. Lowercase always. Consistency matters more than the specific convention.
📄 JSON Everywhere
Request bodies, response bodies, error messages — all JSON. snake_case for keys (customer_id) or camelCase (customerId). Pick one, document it, stick to it.
🔢 HTTP Status Codes
200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests, 500 Internal Server Error. Use the right code. Don't return 200 with an error body.
📄 Error Format
Consistent error structure: { "error": { "code": "invalid_email", "message": "Email must be a valid email address", "field": "email", "request_id": "req_abc123" }}. Machine-readable code + human-readable message + debugging info.
📄 Pagination
Cursor-based pagination for real-time data. Offset-based for static data. Always include: data array, pagination object (has_more, next_cursor, total_count). Never return unbounded lists.

API Versioning

URL versioning: /v1/customers, /v2/customers. Simple, explicit, cache-friendly. Most common approach.

Header versioning: Accept: application/vnd.api+json;version=2. Cleaner URLs, harder to discover. Good for internal APIs.

Deprecation policy: 12-month notice for breaking changes. Sunset headers on deprecated endpoints. Migration guides with before/after examples. Never break existing integrations without warning.

Authentication

API Keys: Simple, fast, good for server-to-server. Pass in header: X-API-Key: sk_live_abc123. Rotateable, scoped, auditable.

OAuth 2.0: Standard for user-delegated access. Authorization code flow for web apps. PKCE for mobile/SPA. Scopes for permissions (read:customers, write:invoices).

JWT: Stateless auth tokens. Signed, not encrypted (use JWE for sensitive claims). Short expiry (15 min) + refresh tokens. Store claims, not user data.

Webhook signatures: Sign payloads with HMAC-SHA256. Verify signature on receipt. Prevents tampering and replay attacks.

Rate Limiting

Return rate limit headers on every request:

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1640995200

Tiered limits: Free: 100/hour, Starter: 1000/hour, Pro: 10000/hour, Enterprise: custom. Align with pricing tiers.

429 response: { "error": { "code": "rate_limit_exceeded", "message": "Rate limit exceeded. Try again in 60 seconds.", "retry_after": 60 } }

API Documentation

OpenAPI (Swagger): Machine-readable API spec. Generates docs, SDKs, and testing tools. Maintain it. Auto-generate from code if possible.

Interactive docs: Try-it-now features. Pre-filled examples. Copy-paste curl commands. Developers should be able to make their first API call in < 5 minutes.

Changelog: Document every change. Breaking, additive, deprecated. Date-stamped. RSS feed. Developers subscribe and stay informed.

SDKs: Official SDKs for top languages (JavaScript, Python, Ruby, PHP, Go). Community SDKs for the rest. SDKs reduce integration time and support tickets.

API Metrics

• Request volume: Calls per day/hour. Growth = adoption.
• Error rate: % of 4xx/5xx responses. Target: < 1% 5xx, < 5% 4xx.
• Latency (P95/P99): Response times. P95 < 200ms, P99 < 500ms.
• SDK adoption: % of API calls via SDK vs. raw HTTP. Higher = better DX.
• Time to first API call: How fast new developers make a successful call. Target: < 10 minutes.
