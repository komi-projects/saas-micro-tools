📊 Monitoring & Observability

Know what's happening before your users do.

The Three Pillars of Observability

📈 Metrics (What's happening?)
Numerical data over time. CPU usage, request latency, error rate, queue depth. Aggregated, queryable, alertable. The "vital signs" of your system. Tools: Prometheus, Datadog, New Relic, Grafana.
📝 Logs (What happened?)
Structured text records of events. "User 123 logged in," "Payment failed for invoice 456." Searchable, filterable, correlated. Tools: ELK Stack, Splunk, Papertrail, LogDNA, CloudWatch Logs.
🔍 Traces (Where did it happen?)
Request flows across services. "User clicked button → API call → DB query → Cache hit → Response." Pinpoint bottlenecks in distributed systems. Tools: Jaeger, Zipkin, Datadog APM, New Relic APM.

Together: Metrics tell you that something is wrong. Logs tell you what happened. Traces tell you where it happened. You need all three.

Key SaaS Metrics to Monitor

Infrastructure:

• CPU utilization (< 70% sustained)
• Memory utilization (< 80%)
• Disk I/O and throughput
• Network latency and packet loss
• Container restart count

Application:

• Request rate (requests/second)
• Response time (P50, P95, P99)
• Error rate (5xx, 4xx, by endpoint)
• Queue depth and processing lag
• Database connection pool usage

Business:

• Signup rate (per hour/day)
• Conversion funnel drop-offs
• Payment success/failure rate
• Active users (DAU/MAU)
• Feature adoption rate

Alerting Strategy

Page (wake someone up): Only for revenue-impacting issues. Payment processing down, signup flow broken, complete outage. If it's not worth waking up at 3 AM, it's not a page.

Notify (Slack/email): For issues that need attention during business hours. Error rate spike, disk > 80%, latency > threshold. Someone should look at it, but not urgently.

Log (dashboard only): For trends and capacity planning. Daily active users, API usage growth, database size. Review weekly in team meetings.

Alert fatigue kills monitoring. If engineers get 50 alerts/day and 49 are false positives, they'll ignore all 50. Start with 5 critical alerts. Add more only when you have signal, not noise.

SLOs, SLIs, and SLAs

SLI (Service Level Indicator): What you measure. "Request latency P95" or "Error rate."

SLO (Service Level Objective): Your target. "P95 latency < 200ms" or "Error rate < 0.1%."

SLA (Service Level Agreement): What you promise customers. "99.9% uptime" or "API response < 500ms." Breach = financial penalty or credits.

Error budget: If your SLO is 99.9% uptime, your error budget is 0.1% downtime (~43 min/month). If you spend your budget, freeze new features and focus on reliability. SRE best practice from Google.

Monitoring Tools Comparison

Datadog: Full-stack observability. Metrics, logs, traces, RUM, security. Expensive but comprehensive. Best for mid-market to enterprise.

Prometheus + Grafana: Open-source metrics and visualization. Free, powerful, requires setup. Best for technical teams with DevOps expertise.

New Relic: APM-focused. Great for application performance monitoring. Pricing based on data ingest. Good for app-centric teams.

Honeycomb: Event-driven observability. Query any attribute, any dimension. Great for debugging complex distributed systems. Pricing per event.

PagerDuty / Opsgenie: Incident management and on-call scheduling. Integrates with all monitoring tools. Essential for paging and escalation.
