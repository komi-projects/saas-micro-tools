🏢 Multi-Tenancy

Architecture patterns for scalable SaaS.

What is Multi-Tenancy?

Multi-tenancy is the architecture pattern where a single software instance serves multiple customers (tenants) while keeping their data and configuration isolated. It's the defining characteristic of SaaS — instead of installing software on each customer's servers, you run one application that hundreds or thousands of customers share.

Done right, multi-tenancy dramatically reduces costs, simplifies updates, and enables economies of scale. Done wrong, it creates security vulnerabilities, performance problems, and data leakage that can destroy your business.

This guide covers the three main multi-tenant architectures, when to use each, and how to implement tenant isolation, performance guarantees, and data security.

Multi-Tenant Architecture Patterns

1. Shared Database, Shared Schema
All tenants share one database and one set of tables. • Tenant isolation: Every table has a tenant_id column; every query filters by tenant_id • Pros: Simplest to implement, easiest cross-tenant analytics, lowest infrastructure cost • Cons: Risk of data leakage if queries forget WHERE tenant_id, harder to scale, noisy neighbor problems • Best for: Startups, small tenants, rapid prototyping, products where tenants don't need isolation guarantees • Example: Most early-stage SaaS products
2. Shared Database, Separate Schema
One database server, but each tenant gets their own schema (namespace). • Tenant isolation: Physical separation at schema level; tenant_123.users vs tenant_456.users • Pros: Better isolation than shared schema, easier compliance, can optimize per tenant • Cons: Complex migrations (run on N schemas), harder cross-tenant queries, schema proliferation • Best for: Mid-market SaaS, compliance requirements, tenants with different needs • Example: Heroku Postgres, some Salesforce implementations
3. Separate Database per Tenant
Each tenant gets their own database instance. • Tenant isolation: Complete physical and logical separation • Pros: Maximum security, no noisy neighbors, tenant-specific optimization, easiest compliance • Cons: Highest cost, complex migrations, hardest cross-tenant analytics, operational overhead • Best for: Enterprise SaaS, regulated industries, customers who demand dedicated infrastructure • Example: Salesforce orgs, some AWS-hosted enterprise products
4. Hybrid (Tier-Based)
Different architectures for different customer tiers. • Free/SMB: Shared schema (cost-efficient, good enough isolation) • Pro/Mid-market: Separate schema (better isolation, compliance) • Enterprise: Separate database (maximum isolation, custom SLAs) • Migration path: Tenants can move up as they grow • Pros: Optimized cost/performance for each segment • Cons: Most complex to implement and maintain • Best for: SaaS with clear free/pro/enterprise tiers

Implementation Considerations

1. Tenant identification:

• Subdomain: tenant.yourapp.com (clean, but requires DNS setup) • Path: yourapp.com/tenant (simpler, but less professional) • Header: X-Tenant-ID in API requests (for API-first products) • JWT claim: Include tenant_id in auth token (cleanest for APIs) • Resolution: Middleware that identifies tenant from request and sets context

2. Data isolation:

• Row-level security: PostgreSQL RLS policies enforce tenant isolation at database level • Query enforcement: Middleware that adds tenant_id to every query • Testing: Automated tests that verify tenant A cannot see tenant B's data • Audit: Log all cross-tenant access attempts (potential security issues)

3. Tenant onboarding:

• Schema provisioning: Automated creation of tenant schemas/databases • Seed data: Default configuration, sample data, templates • Admin setup: First user creation, initial permissions • Customization: Tenant-specific settings, branding, features

4. Performance isolation:

• Resource limits: Per-tenant query limits, rate limits, storage quotas • Query timeouts: Kill queries running too long (prevent one tenant from hogging resources) • Connection pooling: Separate pools per tenant or prioritize by plan tier • Caching: Tenant-scoped cache keys (never share cache between tenants)

Multi-Tenancy Examples

Slack: Workspace Isolation
• Shared infrastructure with workspace-level isolation • Each workspace is a tenant with its own users, channels, and data • Enterprise Grid: Special architecture for large orgs with multiple workspaces • Result: Scales from 2-person startups to 500K-person enterprises
Salesforce: Org-Based Multi-Tenancy
• Each customer gets a "org" — logically separate but on shared infrastructure • Metadata-driven architecture: Custom fields, objects, workflows per org • Trust: Real-time performance and security stats for every org • Result: The gold standard for enterprise SaaS multi-tenancy
Notion: Lightweight Tenancy
• Shared schema with workspace-level data isolation • Users can belong to multiple workspaces (cross-tenant users) • Permission model: Page-level, not just workspace-level • Result: Simple architecture that scales to millions of users

Multi-Tenancy Checklist

☐ Architecture chosen (shared schema, separate schema, separate DB, or hybrid) ☐ Tenant identification method implemented (subdomain, path, header, JWT) ☐ Tenant context middleware implemented (sets tenant for request lifecycle) ☐ Data isolation enforced (RLS or query middleware) ☐ Cross-tenant access tests written (automated security verification) ☐ Tenant onboarding flow automated (schema provisioning, seed data, admin setup) ☐ Tenant offboarding process defined (data export, deletion, account closure) ☐ Resource limits configured (queries, storage, rate limits per tenant) ☐ Query timeouts set (prevent resource hogging) ☐ Tenant-scoped caching implemented (no shared cache keys) ☐ Connection pooling strategy defined ☐ Tenant-specific configuration supported (settings, features, branding) ☐ Cross-tenant analytics pipeline (if needed for product insights) ☐ Tenant migration path defined (shared → separate schema → separate DB) ☐ Backup and recovery per tenant tested ☐ Performance monitoring per tenant (identify noisy neighbors) ☐ Compliance isolation verified (GDPR right to deletion works per tenant) ☐ Custom domain support (enterprise tenants want theirdomain.com) ☐ Tenant-level SSO/OAuth configuration ☐ Billing integration per tenant (metered usage, plan limits) ☐ Documentation: architecture decision record, runbooks, troubleshooting
