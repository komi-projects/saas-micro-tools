One codebase. Thousands of customers. Maximum efficiency.

What is Multi-Tenancy?

Multi-tenancy is an architecture where a single instance of your software serves multiple customers (tenants). Each tenant's data is isolated, but they share the same infrastructure, codebase, and database.

Single-tenant: One deployment per customer. Dedicated servers, dedicated database. Simple, secure, expensive. Good for enterprise with strict compliance.

Multi-tenant: One deployment for all customers. Shared everything. Complex, cost-efficient, scalable. Good for SMB/mid-market SaaS.

Most successful SaaS companies are multi-tenant. AWS, Salesforce, Slack, Stripe — all multi-tenant. The cost savings (5-10x lower infrastructure per customer) and operational simplicity (one deploy affects everyone) are massive.

Multi-Tenancy Models

🔵 Shared Database, Shared Schema
One database, one set of tables. Every table has a tenant_id column. Queries always filter by tenant_id. Simplest model, highest resource efficiency. Risk: one tenant's heavy query can affect others. Mitigation: query limits, resource quotas, read replicas.
🟢 Shared Database, Separate Schema
One database, separate schema per tenant (e.g., tenant_123.users). Better isolation than shared schema. Easy to export one tenant's data. Slightly more complex migrations (apply to all schemas). Good for mid-market with data isolation requirements.
🟡 Separate Database per Tenant
Completely isolated databases. Maximum data isolation. Easier compliance (each tenant's data is physically separate). Higher operational overhead (more connections, more backups, harder migrations). Good for enterprise or regulated industries.
🔴 Hybrid (Tenant-Aware)
Most tenants on shared schema. Large/enterprise tenants on separate databases or schemas. Best of both worlds. Complexity: application must route queries to the right database based on tenant tier. This is what Salesforce and Stripe do.

Data Isolation

The #1 risk of multi-tenancy: Tenant A sees Tenant B's data. This is a catastrophic failure. Prevent it at every layer:

Application layer: Every query must include tenant_id. Use an ORM middleware that automatically injects tenant_id into all queries. Never allow raw SQL without tenant filtering.

API layer: Authenticate and extract tenant from JWT/token. Reject requests that don't include a valid tenant context.

Database layer: Row-level security (RLS) policies in PostgreSQL. Even if application forgets tenant_id, database enforces it.

Testing: Automated tests that verify cross-tenant access is blocked. "As Tenant A, try to access Tenant B's resource" → should return 403.

Tenant Identification

How do you know which tenant a request belongs to?

Subdomain: tenant.yourapp.com. Clean, scalable, requires wildcard SSL. Extract tenant from Host header.

Path: yourapp.com/tenant-slug/page. Simple, works everywhere, less elegant. tenant slug in URL path.

Header/Token: X-Tenant-ID header or JWT claim. Best for APIs. Frontend must include it on every request.

User context: User belongs to tenant. Look up tenant from user ID. Simplest but requires user to be logged in. Doesn't work for public pages.

Resource Management

In shared environments, one tenant can consume disproportionate resources. Implement limits:

Rate limiting: Per-tenant API rate limits. Free: 100/hour, Pro: 10,000/hour, Enterprise: custom. Prevents one tenant from overwhelming the system.

Storage quotas: Max file storage per tenant. Max records per table. Alerts at 80%, hard limit at 100%.

Compute quotas: Max concurrent jobs, max report generation time, max export rows. Kill long-running queries that exceed tenant limits.

Connection pooling: Per-tenant connection limits. Prevents one tenant from exhausting the database connection pool.

Operational Considerations

Migrations: Apply to all tenants simultaneously (shared schema) or sequentially (separate schemas/databases). For large numbers of tenants, use a job queue to apply migrations. Test on a staging tenant first.

Backups: Shared schema: backup whole database. Separate schemas/databases: backup per tenant for compliance, whole database for disaster recovery.

Monitoring: Tag all metrics by tenant_id. Identify noisy tenants. Alert if one tenant generates 50% of errors. Capacity planning per tenant tier.

Data export/deletion: GDPR right to portability and deletion. For shared schema: export tenant's rows, delete tenant's rows. For separate databases: export database, drop database. Automate this — manual exports are error-prone.

When to Choose Each Model

Decision Framework
Shared Schema → Startup, SMB focus, cost-sensitive, no strict data isolation requirements

Separate Schema → Mid-market, some compliance needs, moderate scale (100-10K tenants)

Separate Database → Enterprise, regulated industries (healthcare, finance), strict compliance

Hybrid → Scale from SMB to enterprise. Start shared, migrate large tenants to dedicated resources.
