🧪 A/B Testing

Run experiments that actually move the needle.

What is A/B Testing?

A/B testing (or split testing) is comparing two versions of something to see which performs better. You show version A to 50% of users and version B to the other 50%, then measure which group had better outcomes — more signups, higher conversion, longer retention, etc.

A/B testing removes opinion from product decisions. "I think the blue button will convert better" becomes "The blue button converted 12% better with 95% confidence." Data beats debate.

But A/B testing is easy to do wrong. Running an experiment without enough traffic gives you false positives. Testing too many variables at once makes it impossible to know what changed. Stopping early because "it looks good" invalidates your results. This guide covers how to do it right.

What to A/B Test in SaaS

🎯 Conversion Optimization
• Landing page headlines and CTAs • Pricing page layout and plan names • Signup flow steps and fields • Free trial length (7 vs 14 vs 30 days) • Demo request forms vs. self-serve signup • Social proof placement and type
📈 Activation & Onboarding
• Onboarding sequence order • Empty state designs and copy • Tutorial tooltips vs. video walkthroughs • Progress indicators during setup • Default templates vs. blank slate • Email vs. in-app guidance
💰 Monetization
• Plan pricing ($29 vs $39 vs $49) • Annual discount percentage (15% vs 20% vs 25%) • Paywall timing (day 7 vs day 14) • Upgrade prompts and placement • Feature bundling across plans • Add-on pricing and presentation
🔒 Retention
• Email cadence (daily vs. weekly digests) • In-app notification frequency • Re-engagement campaigns • Feature discovery prompts • Dashboard default views • Mobile app vs. email reminders
📧 Messaging
• Email subject lines • Push notification copy • In-app banner messaging • Error message tone (technical vs. friendly) • Empty state copy and CTAs • Upgrade messaging and urgency

The A/B Testing Process

1. Hypothesis
Start with a belief you can test. Format: "If we [change], then [metric] will [increase/decrease] because [reason]." Bad: "Let's test a new landing page" Good: "If we move social proof above the fold, then signup rate will increase 10% because prospects need trust before giving their email" One hypothesis per test. Testing multiple changes at once = multivariate testing (needs way more traffic).
2. Design the Experiment
• Define the primary metric (the one that decides winner) • Define guardrail metrics (things that shouldn't get worse) • Calculate sample size (see below) • Determine runtime (at least 1 full business cycle, ideally 2) • Define traffic split (usually 50/50) • Document what success looks like
3. Build and QA
• Implement both variants • Ensure random assignment (same user always sees same variant) • Test tracking fires correctly for both • Verify no flicker or layout shift • Check mobile and desktop • Run A/A test first (both variants are the same) to verify your tooling
4. Run the Test
• Don't peek at results daily (statistical peeking inflates false positives) • Run for the planned duration even if early results look good • Monitor guardrail metrics • If something breaks (e.g., variant crashes), stop immediately • Document any external events that could affect results (holidays, launches, outages)
5. Analyze Results
• Check statistical significance (p < 0.05 minimum, p < 0.01 better) • Check statistical power (should be > 80%) • Segment results (did mobile users respond differently?) • Look at guardrail metrics (did we hurt activation while improving conversion?) • Calculate confidence intervals (not just point estimates) • Document learnings regardless of outcome
6. Implement Winner
• Ship the winning variant to 100% • Monitor for 1-2 weeks post-shipping • Document the learnings in your experiment log • Share results with the team (even failed tests teach something) • Plan follow-up tests (winning variant can often be improved further)

Sample Size Calculator

You need enough users in each variant to detect a meaningful difference. Too few = false negatives (missed wins). Too many = wasted time.

Minimum detectable effect (MDE): The smallest improvement you care about. If your baseline conversion is 5%, an MDE of 20% means you want to detect changes of ±1 percentage point (5% → 6% or 5% → 4%).

Formula (simplified):

n = 16 × σ² / δ²

Where:
• n = sample size per variant
• σ = standard deviation of metric (for conversion, σ = √(p×(1-p)) where p is baseline rate)
• δ = minimum detectable effect

Example: Landing Page CTA
Baseline conversion: 5%
MDE: 20% relative (5% → 6%)
σ = √(0.05 × 0.95) = 0.218
δ = 0.01 (1 percentage point)
n = 16 × (0.218)² / (0.01)² = 16 × 0.0475 / 0.0001 = 7,600 per variant

Total: ~15,200 visitors for a 50/50 split.
At 1,000 visitors/day: ~15 days.
At 10,000 visitors/day: ~2 days.

Shortcuts:

• Use an online calculator (Optimizely, VWO, or Evan Miller's)
• For conversion rates: rule of thumb is 100 conversions per variant minimum
• For revenue: you need way more (higher variance)
• Smaller MDE = larger sample needed. Be realistic about what you can detect.

Statistical Significance

Statistical significance tells you if the observed difference is real or just random noise. p-value < 0.05 means there's less than a 5% chance the difference is due to luck.

Common mistakes:

❌ Peeking: Checking results daily and stopping when it "looks significant." This massively inflates false positives. Decide your runtime upfront and stick to it.
❌ Multiple comparisons: Testing 20 metrics and reporting the 1 that "worked." With 20 metrics, you'd expect 1 false positive by chance. Pick 1 primary metric before running the test.
❌ Low power: Stopping because "there's no difference" when you never had enough traffic to detect one. Always calculate required sample size beforehand.
❌ Segment hunting: "It didn't work overall, but for users in California on Tuesdays it was +50%!" This is cherry-picking. If you didn't pre-define the segment, the result is likely noise.

Best practices:

✅ Set sample size and runtime before starting
✅ Define 1 primary metric and 2-3 guardrails
✅ Run for at least 1 full business cycle (7 days minimum)
✅ Use Bayesian methods if you want probability of winner (not just p-values)
✅ Always report confidence intervals, not just "it won by 15%"

A/B Testing Tools

Full-stack experimentation:

• Optimizely: Enterprise-grade. Web, mobile, server-side. Expensive but comprehensive.
• VWO (Visual Website Optimizer): Good for marketers. Visual editor, heatmaps, surveys. Mid-market pricing.
• Google Optimize (discontinued 2023): Was free. Many teams moved to VWO or Optimizely.
• LaunchDarkly: Feature flags + experimentation. Developer-friendly.
• Split: Feature flags with built-in A/B testing and analytics.
• PostHog: Open-source product analytics + feature flags + A/B testing. Great for startups.

Build your own:

• Random assignment via feature flag system • Event tracking in your analytics (Segment, Mixpanel, Amplitude) • Analysis in SQL or Python (scipy.stats.chi2_contingency for conversion, t-test for continuous metrics) • Good for: teams with strong data engineering, limited budgets, or complex custom needs

☐ Guardrail metrics defined
