⚡ Performance

Fast apps win. Slow apps churn.

Why Performance Matters

Performance is a feature. Amazon found that every 100ms of latency cost 1% in revenue. Google found that 53% of mobile users abandon sites that take > 3 seconds to load. In SaaS, slow dashboards, laggy reports, and unresponsive UIs drive churn.

Users don't care about your tech stack. They care about speed. A fast app feels premium. A slow app feels broken.

Core Web Vitals (Google's Performance Standard)

LCP — Largest Contentful Paint
How fast the main content loads. Target: < 2.5 seconds. Measure with Chrome DevTools, PageSpeed Insights, or Web Vitals extension. Optimize: compress images, use CDN, lazy load, server-side rendering, eliminate render-blocking resources.
INP — Interaction to Next Paint
How responsive your app feels to user input. Target: < 200ms. Optimize: break up long JavaScript tasks, use web workers, debounce expensive operations, virtualize long lists, avoid layout thrashing.
CLS — Cumulative Layout Shift
Visual stability. Do elements jump around as the page loads? Target: < 0.1. Optimize: set explicit dimensions on images/videos, reserve space for dynamic content, avoid inserting content above existing content, use font-display: swap carefully.

Backend Performance

Database:

• Query optimization: EXPLAIN ANALYZE slow queries. Add indexes. Avoid SELECT *. Use pagination.
• Connection pooling: Don't open a new connection per request. Use PgBouncer or equivalent.
• N+1 prevention: Eager load associations. Use DataLoader pattern for GraphQL.
• Read replicas: Offload reporting queries to replicas.

API:

• Response time: P95 < 200ms for simple reads, < 500ms for complex queries.
• Payload size: Don't return 10MB JSON. Paginate, filter fields, compress responses (gzip/Brotli).
• Caching: Cache GET requests. ETags for conditional requests. CDN for static assets.
• Rate limiting: Protect your API from abuse. Return 429 with Retry-After header.

Async Work:

• Move slow operations to background jobs (email, exports, reports, webhooks).
• Return 202 Accepted with a job ID for async endpoints.
• Use queues with retry logic and dead letter queues.
• Set max job runtime to prevent runaway tasks.

Frontend Performance

Bundle Size:

• Code splitting: Load only the JavaScript needed for the current page. React.lazy(), dynamic imports.
• Tree shaking: Remove unused code. Use ES modules, avoid importing entire libraries.
• Dependency audit: Regularly check bundle size with webpack-bundle-analyzer. A 1MB+ bundle is a problem.

Rendering:

• Server-side rendering (SSR): Faster initial paint, better SEO. Next.js, Nuxt, SvelteKit.
• Static site generation (SSG): Pre-render pages at build time. Instant load for marketing pages, docs.
• Virtualization: For lists with 1000+ items, only render visible items. react-window, react-virtualized.

Assets:

• Images: WebP format, responsive images (srcset), lazy loading, compression. Use a service like Cloudinary or Imgix.
• Fonts: Subset fonts to only needed characters. font-display: swap. Preload critical fonts.
• CSS: Critical CSS inline for above-the-fold content. Defer non-critical CSS.

Performance Budgets

Set limits and enforce them in CI/CD:

• JavaScript: < 200KB gzipped for initial load
• Images: < 500KB total for above-the-fold
• Fonts: < 100KB total
• API response: < 100KB for standard endpoints
• Total page weight: < 1MB for landing pages
• LCP: < 2.5s
• TTFB (Time to First Byte): < 200ms

Tools: Lighthouse CI, bundlesize, performance budgets in webpack. Fail the build if budget is exceeded.

Measuring Performance

Lab data (controlled testing):

• Lighthouse: Chrome DevTools built-in. Scores 0-100. Good baseline.
• WebPageTest: Detailed waterfall, filmstrip, comparison across locations/devices.
• k6/Artillery: API performance testing. Load testing at scale.

Field data (real users):

• Google Analytics 4: Core Web Vitals report.
• Sentry Performance: Frontend and backend transaction tracing.
• Datadog/Raygun/New Relic: Real user monitoring (RUM), APM.
• Web Vitals library: Measure LCP, INP, CLS in production, send to analytics.

Measure both. Lab data shows what could happen. Field data shows what does happen.
