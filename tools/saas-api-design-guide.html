🔌 API Design

Build APIs developers love.

Why API Design Matters

Your API is your product's interface to the world. Developers judge your entire company by your API — its consistency, documentation, reliability, and developer experience. A great API turns customers into advocates who build on your platform, integrate your product into their workflows, and recommend you to other developers.

The best SaaS APIs share common traits: they're predictable (same patterns everywhere), well-documented (clear examples, not just reference), versioned (changes don't break existing integrations), and developer-friendly (helpful errors, SDKs, sandboxes). Stripe set the gold standard — their API is so good that "Stripe-like" became the benchmark for developer experience.

This guide covers the principles, patterns, and practices that make APIs developers love to use.

API Design Principles

1. Consistency
Same patterns everywhere. • Naming: Use snake_case or camelCase consistently • Structure: Same response format for all endpoints • Methods: POST for create, GET for read, PUT/PATCH for update, DELETE for remove • Errors: Same error format across all endpoints • Pagination: Same pagination params (limit, offset or cursor) • Filtering: Same query parameter patterns (?status=active&sort=-created_at)
2. Predictability
Developers should guess correctly. • URL structure: /resources/{id}/subresources (RESTful nesting) • Status codes: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Rate Limited, 500 Server Error • Error format: { "error": { "type": "", "message": "", "code": "", "param": "" } } • Response format: { "data": {}, "meta": {}, "links": {} } • Idempotency: POST with Idempotency-Key header for safe retries
3. Discoverability
Easy to learn, hard to forget. • Documentation: Interactive docs with try-it-now (Swagger/OpenAPI) • Examples: Every endpoint has request/response examples • SDKs: Official libraries for popular languages (JavaScript, Python, Ruby, Go) • Postman collection: Import and start testing in 30 seconds • Changelog: Document every API change with migration guide • Blog: API tutorials, use cases, best practices
4. Reliability
It works, always. • Uptime SLA: 99.9% or 99.99% with financial backing • Rate limits: Clear headers (X-RateLimit-Remaining, X-RateLimit-Reset) • Timeouts: Document expected response times, set client timeouts • Retries: Document retry strategy (exponential backoff, max retries) • Idempotency: Safe to retry without double-charging or double-creating • Status page: Public status page with incident history
5. Evolvability
Change without breaking. • Versioning: URL path (/v1/, /v2/) or header (Accept: application/vnd.api+json;version=2) • Deprecation: 6-12 month notice before removing endpoints • Sunsetting: Gradual deprecation with warnings, then errors, then removal • Backwards compatibility: Never change existing field types or meanings • Additive changes: Only add new fields, never remove or rename existing • Migration guides: Step-by-step for major version changes

REST API Best Practices

URL Structure:

GET /v1/customers # List customers GET /v1/customers/123 # Get specific customer POST /v1/customers # Create customer PUT /v1/customers/123 # Update customer (full) PATCH /v1/customers/123 # Update customer (partial) DELETE /v1/customers/123 # Delete customer GET /v1/customers/123/orders # List customer's orders

Response Format:

{ "data": { "id": "cust_123", "name": "Acme Corp", "email": "admin@acme.com", "created_at": "2024-01-15T10:30:00Z", "metadata": { "plan": "pro", "mrr": 299 } }, "meta": { "request_id": "req_abc123", "timestamp": "2024-01-15T10:30:00Z" } }

Error Format:

{ "error": { "type": "invalid_request", "message": "Invalid email format", "code": "email_invalid", "param": "email", "request_id": "req_abc123" } }

Pagination:

# Offset-based (simple) GET /v1/customers?limit=20&offset=40 # Cursor-based (better for large datasets) GET /v1/customers?limit=20&cursor=eyJpZCI6MTIzfQ== # Response { "data": [...], "pagination": { "has_more": true, "next_cursor": "eyJpZCI6MTI0fQ==", "total_count": 1500 } }

Filtering & Sorting:

GET /v1/customers?status=active&plan=pro&sort=-created_at # status=active: filter by status # plan=pro: filter by plan # sort=-created_at: sort by created_at descending (- prefix)

Rate Limiting:

HTTP/1.1 429 Too Many Requests X-RateLimit-Limit: 100 X-RateLimit-Remaining: 0 X-RateLimit-Reset: 1640995200 Retry-After: 60 { "error": { "type": "rate_limit_exceeded", "message": "Rate limit exceeded. Try again in 60 seconds.", "code": "rate_limit" } }

API Documentation

1. OpenAPI Spec: Machine-readable API definition. Generate docs, SDKs, and tests from one source of truth.

2. Interactive docs: Swagger UI or ReDoc where developers can try endpoints live. Include authentication helpers.

3. Getting started guide: "Your first API call in 5 minutes." Copy-paste ready code in multiple languages.

4. Authentication: Clear instructions for API keys, OAuth, or JWT. Include token generation examples.

5. Webhooks: Document event types, payload formats, retry logic, and signature verification.

6. SDKs: Official libraries for JavaScript/TypeScript, Python, Ruby, PHP, Go, Java. Community SDKs are nice, official ones are expected.

7. Changelog: Every API change documented with date, impact, and migration steps. RSS feed for subscribers.

8. Status page: Public page showing API uptime, incident history, and planned maintenance.

API Security

1. Authentication: API keys for simple cases, OAuth 2.0 for user delegation, JWT for stateless auth. Never pass tokens in URL params.

2. HTTPS only: No HTTP, ever. HSTS headers. TLS 1.2+.

3. Rate limiting: Per-key limits to prevent abuse. Different limits for different plans.

4. Input validation: Validate everything: types, ranges, formats, lengths. Sanitize to prevent injection.

5. Idempotency: Idempotency-Key header for POST requests. Prevents double-charges, double-creates, double-anything.

6. Webhook signatures: Sign webhook payloads with HMAC. Document signature verification in every language.

7. CORS: Allow browser-based API calls from approved domains. Don't allow * in production.

8. Audit logging: Log all API calls with IP, key, endpoint, and timestamp. Retain for security investigation.

API Design Checklist

☐ RESTful URL structure (nouns, not verbs; plural resources) ☐ Consistent naming convention (snake_case or camelCase) ☐ Standard HTTP methods (GET, POST, PUT, PATCH, DELETE) ☐ Proper status codes (200, 201, 400, 401, 404, 429, 500) ☐ Consistent error format across all endpoints ☐ Response envelope (data, meta, pagination) ☐ Pagination strategy (cursor-based for large datasets) ☐ Filtering and sorting (query parameters) ☐ Rate limiting with clear headers ☐ Idempotency support for mutating operations ☐ API versioning strategy (URL or header) ☐ Deprecation policy (6-12 month notice) ☐ OpenAPI specification generated ☐ Interactive documentation (Swagger UI, ReDoc) ☐ Getting started guide (5-minute tutorial) ☐ Authentication documentation (API keys, OAuth) ☐ SDKs for major languages (JS, Python, Ruby, Go) ☐ Webhook documentation (events, payloads, retries) ☐ Changelog with RSS feed ☐ Public status page ☐ HTTPS only, TLS 1.2+ ☐ Input validation and sanitization ☐ Webhook signature verification ☐ CORS configured correctly ☐ Audit logging enabled ☐ Performance monitoring (p50, p95, p99 response times) ☐ SDK code examples in documentation ☐ Error handling guide for developers ☐ Postman/Insomnia collection available ☐ API playground/sandbox environment
