Ship safely, test in production, control rollouts.

What Are Feature Flags?

Feature flags (also called feature toggles) are a software development technique that lets you turn features on or off without deploying new code. They decouple deployment from release — you can deploy code to production but hide it from users until you're ready.

Think of them as remote controls for your product. Instead of shipping a feature and hoping it works, you ship the code, test it with a small group, measure the impact, and gradually roll it out. If something goes wrong, you turn it off instantly — no rollback, no hotfix, no downtime.

The best SaaS companies use feature flags for everything: gradual rollouts, A/B testing, kill switches, beta programs, and entitlements. This guide covers how to implement feature flags, when to use them, and best practices for managing them at scale.

Types of Feature Flags

1. Release Toggles
Hide incomplete features from production. • Use case: Deploy code for a feature still in development • Lifecycle: Short-term, removed after feature launches • Example: New dashboard layout — code deployed, flag off until design is finalized • Rule of thumb: If it's not ready for customers, hide it behind a flag
2. Experiment Toggles
Run A/B tests and measure impact. • Use case: Test two versions of a feature with real users • Lifecycle: Short-term, removed after experiment concludes • Example: Test new checkout flow — 50% see version A, 50% see version B • Measure: Conversion, engagement, revenue, support tickets
3. Kill Switches
Instantly disable problematic features. • Use case: Turn off a feature causing errors or performance issues • Lifecycle: Long-term, permanent safety mechanism • Example: Disable a new integration if its API goes down • Key: Must work instantly, no code deploy required
4. Canary Toggles
Gradually roll out to a percentage of users. • Use case: Reduce risk by exposing features to small groups first • Lifecycle: Short-term, ramped to 100% or rolled back • Example: Start at 1%, then 5%, 10%, 50%, 100% if metrics look good • Automatic: Set error rate thresholds for auto-rollback
5. Entitlement Toggles
Control access based on plan or user type. • Use case: Premium features available only to paying customers • Lifecycle: Long-term, permanent part of product packaging • Example: "Advanced reporting" flag — on for Pro/Enterprise, off for Basic • Benefit: No code changes needed to adjust plan features
6. Ops Toggles
Operational controls for system behavior. • Use case: Adjust system behavior without code changes • Lifecycle: Long-term, infrastructure management • Example: Circuit breaker for external API calls, rate limiting thresholds • Use sparingly: These are for ops, not product features

Feature Flag Architecture

1. Simple approach (start here):

• Config file: Store flags in a JSON config file • Environment variables: One flag per env var • Database: Simple table with flag name, status, and targeting rules • Pros: Fast to implement, no external dependencies • Cons: Requires redeploy to change, no targeting, hard to manage at scale

2. Managed service (scale here):

• LaunchDarkly: Market leader, full feature set, $8.33/seat/month • Split: Strong experimentation, analytics, $33/seat/month • Flagsmith: Open-source, self-hosted option, free tier • Unleash: Open-source, popular in enterprise, free tier • GrowthBook: Open-source, strong experimentation focus, free tier • PostHog: Product analytics + feature flags, generous free tier

3. Build vs. buy:

• Build: If you need simple on/off flags and have 1-2 teams • Buy: If you need targeting, experiments, analytics, or have 3+ teams • Hybrid: Build simple toggles, buy for experiments and canary releases • Cost of building: 2-4 weeks engineering, ongoing maintenance • Cost of buying: $50-500/month depending on seats and features

Feature Flag Best Practices

1. Naming: Use descriptive names with namespaces. "new-checkout-flow" not "flag_7". Include team prefix: "billing-new-checkout-flow".

2. Cleanup: Remove flags after they're no longer needed. "Permanent" flags should be rare. Schedule quarterly flag audits.

3. Testing: Test both flag on and flag off paths. A feature behind a flag that breaks when off is a hidden bug.

4. Documentation: Document what each flag does, who owns it, and when it should be removed. "Flag registry" in your wiki.

5. Access control: Not everyone should toggle production flags. Restrict to on-call engineers and product managers.

6. Monitoring: Track flag changes in your alerting system. "Flag X was turned off at 2:15 PM" should be visible.

7. Fallbacks: If flag service is down, default to safe behavior. Usually "off" is safer than "on" for new features.

8. Avoid flag explosion: Don't create flags for everything. Use for: risky changes, experiments, entitlements, kill switches. Don't use for: trivial UI changes, bug fixes, config values.

Feature Flag Examples

Netflix: Gradual Codec Rollout
Netflix uses feature flags to roll out new video codecs gradually. • Start: 1% of devices, measure buffer health and error rates • Expand: 5%, 10%, 25%, 50%, 100% if metrics stable • Kill switch: If error rate > 0.1%, instant rollback to 0% • Result: Billions of streaming hours with minimal disruption
Etsy: Continuous Deployment
Etsy deploys 50+ times per day using feature flags. • All changes behind flags by default • Engineers toggle on for themselves first • Then team, then 1% of users, then full rollout • Result: Fast iteration with production safety
Stripe: Kill Switches
Stripe maintains kill switches for all third-party integrations. • External API down? Toggle off instantly. • Payment processor issue? Route traffic to backup. • New fraud rule causing false positives? Disable immediately. • Result: 99.999% uptime despite dependency failures

Feature Flag Checklist

☐ Feature flag system chosen (build, open-source, or managed) ☐ SDK integrated into your application ☐ Flag naming convention established (team-feature-name) ☐ Flag registry created (documentation of all flags) ☐ Cleanup process defined (quarterly audits, removal deadlines) ☐ Access control configured (who can toggle what) ☐ Monitoring set up (flag change alerts) ☐ Fallback behavior defined (default to safe state) ☐ Testing strategy for both flag states (on and off) ☐ Kill switches implemented for critical features ☐ Canary rollout process defined (1% → 5% → 10% → 50% → 100%) ☐ A/B test framework configured (experiment toggles) ☐ Entitlement toggles set up for plan-based features ☐ Flag change audit log enabled (who changed what, when) ☐ Performance impact measured (flag evaluation latency) ☐ Runbook created for flag-related incidents ☐ Team trained on flag usage and best practices ☐ Flag expiration dates set (auto-remind after 30 days)
