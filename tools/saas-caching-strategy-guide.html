Performance at scale, costs under control.

Why Caching is Essential for SaaS

Every database query, API call, and computation costs time and money. Caching stores frequently accessed data in fast, inexpensive memory so you don't repeat expensive work. A well-designed caching strategy can reduce database load by 80%, cut response times from seconds to milliseconds, and handle 10x traffic spikes without breaking a sweat.

The best SaaS companies cache at multiple layers: browser cache for static assets, CDN for global delivery, Redis for application state, and database query cache for repeated lookups. Each layer serves a different purpose and together they create a resilient, performant system.

This guide covers caching strategies for every layer of your SaaS stack, from browser to database, with specific implementation patterns and trade-offs.

Caching Layers

1. Browser Cache
Static assets cached in the user's browser. • Cache-Control headers: max-age=31536000 for immutable assets • ETags: Conditional requests for updated resources • Service workers: Offline-first apps, background sync • LocalStorage/IndexedDB: User preferences, draft data • Pros: Zero server load, instant access, works offline • Cons: Limited storage (~5-50MB), no shared state between users
2. CDN (Content Delivery Network)
Global edge servers cache content close to users. • Static assets: Images, CSS, JS, fonts (Cloudflare, Fastly, AWS CloudFront) • Dynamic content: Cache HTML at edge with stale-while-revalidate • API responses: Cache GET requests at edge with short TTL • Purging: Invalidate cache on deploy or content update • Pros: Global scale, DDoS protection, reduced origin load • Cons: Cache invalidation complexity, cost at high scale
3. Application Cache (Redis/Memcached)
In-memory data store for hot data. • Session storage: User sessions, auth tokens • Rate limiting: Sliding window counters • Feature flags: Fast flag evaluation • Computed data: Dashboard metrics, aggregations • Queue buffers: Temporary job storage • Pros: Sub-millisecond access, rich data structures, pub/sub • Cons: Memory limits, persistence trade-offs, eviction complexity
4. Database Query Cache
Cache repeated database queries. • PostgreSQL: Query plan cache, prepared statements • Application-level: Cache ORM query results • Materialized views: Pre-computed aggregations • Connection pooling: Reuse connections, reduce overhead • Pros: Transparent to application, automatic invalidation • Cons: Limited to query results, per-connection in some databases
5. Full-Page Cache
Cache entire rendered pages. • Varnish/Squid: HTTP reverse proxy cache • Edge functions: Cache at CDN with personalization • Fragment caching: Cache page components independently • Pros: Dramatic speedup for public pages, reduced server load • Cons: Stale content, hard to personalize, invalidation complexity

Cache Invalidation Strategies

1. TTL (Time to Live):

• Set expiration time when writing to cache • Simplest approach, good for data that changes predictably • Example: Cache user profile for 5 minutes, product catalog for 1 hour • Stale-while-revalidate: Serve stale data while refreshing in background • Risk: Serving outdated data during TTL window

2. Write-through:

• Update cache when database is updated • Cache is always fresh, but writes are slower • Best for: Data that must be consistent (user permissions, feature flags) • Trade-off: Write latency increases by cache round-trip

3. Write-behind (async):

• Write to cache immediately, update database asynchronously • Fastest writes, risk of data loss if cache fails before database update • Best for: High-write throughput, eventually consistent acceptable • Requires: Queue (Redis, Kafka) to buffer database writes

4. Cache-aside (lazy loading):

• Check cache first, load from DB on miss, write to cache • Most common pattern, simple to implement • Risk: Cache stampede (thundering herd) on expiration • Mitigation: Staggered TTLs, request coalescing, circuit breakers

5. Event-based invalidation:

• Listen to database changes (CDC), invalidate affected cache keys • Most precise invalidation, no stale data • Requires: Change data capture (Debezium, logical replication) • Best for: Complex caching where TTL is too imprecise

Redis Caching Patterns

1. String cache:

# Cache user profile SET user:123 "{name: 'John', plan: 'pro'}" EX 300 # Read GET user:123 # Delete on update DEL user:123

2. Hash cache:

# Cache object with multiple fields HSET product:456 name "Widget" price 99 stock 50 EXPIRE product:456 600 # Read specific field HGET product:456 price

3. Set cache:

# Cache user permissions SADD permissions:123 "read" "write" "delete" EXPIRE permissions:123 3600 # Check permission SISMEMBER permissions:123 "write"

4. Sorted set for leaderboards:

# Cache top customers by MRR ZADD leaderboard:mrr 5000 "customer:1" 3200 "customer:2" ZREVRANGE leaderboard:mrr 0 9 WITHSCORES

5. Pub/sub for invalidation:

# Subscribe to invalidation channel SUBSCRIBE cache-invalidate # Publish invalidation PUBLISH cache-invalidate "user:123"

Caching Mistakes

1. No cache: Querying the database for every request. The database is 1000x slower than memory. Cache aggressively.

2. Over-caching: Caching everything including dynamic, user-specific data. Creates stale data issues and complexity.

3. Cache stampede: Thousands of requests hit cache simultaneously when it expires, all querying the database. Use staggered TTLs, request coalescing, or circuit breakers.

4. No invalidation: Setting long TTLs without a way to invalidate early. Users see stale data for hours. Always have a way to force refresh.

5. Cold start: Deploying cache with empty data, causing database overload. Warm cache on deploy with common queries.

6. Ignoring eviction: Redis evicts keys when memory is full. Configure eviction policy (allkeys-lru, volatile-lru) and monitor eviction rate.

7. Single cache server: One Redis instance is a single point of failure. Use Redis Cluster or Sentinel for HA.

Caching Strategy Checklist

☐ Browser cache configured (Cache-Control, ETags, service workers) ☐ CDN configured (static assets, API responses with short TTL) ☐ Redis/Memcached deployed (clustered for HA) ☐ Connection pooling configured (PgBouncer, ProxySQL) ☐ Cache layers documented (what goes where, why) ☐ TTL strategy defined (per data type, per endpoint) ☐ Invalidation strategy chosen (TTL, write-through, event-based) ☐ Cache stampede protection (staggered TTLs, request coalescing) ☐ Cold start plan (cache warming on deploy) ☐ Redis eviction policy set (allkeys-lru recommended) ☐ Memory monitoring configured (eviction rate, memory usage) ☐ Hit rate monitoring (cache efficiency) ☐ Cache key naming convention (resource:id:field) ☐ Serialization format chosen (JSON, MessagePack, Protobuf) ☐ Compression enabled (for large values) ☐ Circuit breaker configured (fail fast when cache is down) ☐ Cache debugging tools (clear cache, inspect keys) ☐ Performance benchmarks (before/after cache implementation) ☐ Cost analysis (cache infrastructure vs. database savings) ☐ Disaster recovery (cache rebuild procedures)
