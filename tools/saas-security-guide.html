🔒 Security

Protect your product, your customers, and your reputation.

Security is Not a Feature — It's the Foundation

In SaaS, security breaches don't just cost money — they destroy trust. One leaked customer database, one unauthorized access, one ransomware attack, and your reputation is damaged for years. Security isn't something you bolt on later; it's something you build in from day one, layer by layer, until it becomes part of your culture.

The best SaaS companies treat security as a competitive advantage. They publish security whitepapers, share penetration test results, maintain bug bounty programs, and build trust through transparency. Their customers sleep better knowing their data is protected by people who take it seriously.

This guide covers the essential security practices every SaaS company needs — from authentication to encryption to incident response. Not everything applies on day one, but knowing the full picture helps you build incrementally toward a secure product.

Authentication & Authorization

1. Password Security
Don't roll your own — use proven methods. • Hashing: bcrypt, scrypt, or Argon2 (never MD5, SHA1, or plain text) • Salting: Unique salt per password, stored with the hash • Complexity: Minimum 8 characters, but length > complexity (passphrases > passwords) • Breach detection: Check passwords against Have I Been Pwned database on signup • MFA: Enforce for admins, offer for all users (TOTP, WebAuthn, SMS fallback) • Password resets: Secure tokens (random, time-limited, single-use), sent via email
2. Session Management
Secure tokens, short lifetimes, proper revocation. • JWT: Signed (not encrypted), short expiry (15-60 min), refresh tokens for long sessions • Cookies: HttpOnly, Secure, SameSite=Strict, path=/ • Session storage: Redis for server-side sessions, signed cookies for stateless • Revocation: Instant logout everywhere, session invalidation on password change • Idle timeout: 30 minutes of inactivity → re-authentication • Concurrent sessions: Limit or allow, but show active sessions to user
3. Role-Based Access Control (RBAC)
Define who can do what. • Roles: Owner, Admin, Member, Viewer (customize for your product) • Permissions: Granular (create, read, update, delete, export, invite) • Resources: What objects can they access? (own data, team data, org data) • Enforcement: Check permissions at API gateway AND service layer (defense in depth) • Default deny: If permission isn't explicitly granted, it's denied • Audit: Log every permission check failure (potential attack indicator)
4. OAuth & SSO
Let customers use their identity provider. • OAuth 2.0: Standard for delegated authorization (Google, GitHub, Microsoft login) • SAML 2.0: Enterprise SSO (Okta, Azure AD, OneLogin) • OIDC: Modern standard built on OAuth 2.0, preferred for new implementations • SCIM: Automatic user provisioning/deprovisioning from identity provider • Security: Validate tokens properly, use PKCE for SPA/mobile, verify state parameter • UX: "Sign in with Google" reduces friction, SSO reduces IT friction

Data Protection

1. Encryption at rest:

• Database: Enable TDE (Transparent Data Encryption) or disk-level encryption • Files: Encrypt uploads before storing (AES-256) • Backups: Encrypt all backups, test restore procedures regularly • Keys: Use KMS (AWS KMS, Google Cloud KMS, Azure Key Vault) — don't hardcode keys • Key rotation: Automatic rotation every 90 days for managed keys

2. Encryption in transit:

• TLS 1.2+ everywhere: No HTTP, ever. HSTS headers with long max-age. • Certificate management: Automated with Let's Encrypt or managed certs • API security: mTLS for service-to-service communication in microservices • Internal traffic: Encrypt even internal service communication (zero trust) • Downgrade prevention: TLS 1.3 where possible, disable weak cipher suites

3. Data classification:

• Public: Marketing content, pricing, documentation • Internal: System configs, non-sensitive logs • Confidential: Customer data, business metrics, API keys • Restricted: PII, payment data, health records (PCI, HIPAA) • Labeling: Tag data by classification, apply controls accordingly

4. Data retention & deletion:

• Retention policy: Define how long you keep each data type • Automated deletion: Cron jobs or lifecycle policies for old data • Right to deletion: GDPR Article 17 — delete user data on request within 30 days • Backup cleanup: Ensure deleted data doesn't persist in backups forever • Audit: Log all deletions for compliance

Vulnerability Management

1. Dependency scanning:

• Automated: Snyk, Dependabot, GitHub Advanced Security • CI/CD integration: Block builds with critical/high vulnerabilities • Update cadence: Weekly review of new vulnerabilities, monthly updates • Pin versions: Lock files (package-lock.json, yarn.lock, poetry.lock) for reproducibility

2. Static analysis:

• SAST tools: SonarQube, CodeQL, Semgrep • Rules: SQL injection, XSS, insecure deserialization, hardcoded secrets • Integration: Pre-commit hooks and CI checks • False positives: Tune rules to reduce noise, but don't disable security rules

3. Penetration testing:

• Annual: Hire external security firm for full penetration test • Quarterly: Internal security review of new features • Bug bounty: HackerOne, Bugcrowd, or self-managed program • Scope: Define what's in scope (web app, API, mobile) and out of scope • Remediation: 30-day SLA for critical, 90-day for high findings

4. Infrastructure security:

• Network: VPC, security groups, no public databases • Containers: Non-root users, minimal base images, vulnerability scanning • Secrets: Vault, AWS Secrets Manager, or 1Password — never in code or env vars in repos • IAM: Least privilege, no shared credentials, MFA for all admin access • Patching: Automated OS patching, container image updates monthly

Incident Response

1. Preparation:

• Incident response plan: Documented, tested, reviewed quarterly • Response team: Primary, secondary, and executive contacts • Communication templates: Customer notification, status page update, regulatory notice • Forensics: Logging configured for investigation (audit logs, access logs, change logs) • Insurance: Cyber liability insurance with appropriate coverage limits

2. Detection:

• Automated: SIEM rules, anomaly detection, threshold alerts • Manual: Customer reports, employee reports, security researcher reports • Triage: Classify severity (P0 data breach, P1 unauthorized access, P2 suspicious activity) • Escalation: Who to wake up, when, and how

3. Response:

• Contain: Isolate affected systems, revoke compromised credentials, block malicious IPs • Eradicate: Remove attacker access, patch vulnerabilities, rotate keys • Recover: Restore from clean backups, verify system integrity, resume operations • Communicate: Notify customers within 72 hours (GDPR), publish incident report

4. Post-incident:

• Postmortem: Timeline, root cause, impact assessment, lessons learned • Action items: Specific, assigned, with deadlines • Share: Publish redacted postmortem publicly (builds trust) • Update: Revise incident response plan based on learnings
