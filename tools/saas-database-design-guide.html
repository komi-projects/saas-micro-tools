Why Database Design Matters in SaaS

Database design is the foundation of your SaaS product. Get it wrong early, and every subsequent feature becomes harder to build, slower to query, and more expensive to scale. Get it right, and you have a platform that grows smoothly from 10 customers to 10,000 without rewrites.

SaaS databases face unique challenges: multi-tenancy (keeping customer data separate), scalability (handling millions of rows per tenant), compliance (GDPR, SOC2, HIPAA), and performance (sub-100ms queries under load). The choices you make in month 1 affect you in year 5.

This guide covers database design patterns for SaaS, from multi-tenant architectures to indexing strategies to scaling approaches that work from startup to enterprise.

Multi-Tenant Architectures

1. Shared Database, Shared Schema
All tenants in one database, one set of tables. • Tenant ID column: Every table has a tenant_id foreign key • Query pattern: WHERE tenant_id = ? on every query • Pros: Simplest to implement, easiest to query across tenants, lowest infrastructure cost • Cons: Harder to scale (one giant database), noisy neighbor risk, complex compliance isolation • Best for: Early-stage SaaS, small tenants, rapid prototyping • Example: Most SaaS startups start here
2. Shared Database, Separate Schema
One database, but each tenant has their own schema (namespace). • Schema per tenant: tenant_1234.users, tenant_1234.orders • Pros: Better isolation than shared schema, easier compliance, can optimize per tenant • Cons: More complex migrations (apply to N schemas), harder cross-tenant analytics • Best for: Mid-size SaaS, compliance requirements, varied tenant sizes • Example: Heroku Postgres, some Salesforce implementations
3. Separate Database per Tenant
Each tenant gets their own database instance. • Complete isolation: No shared resources, maximum security • Pros: Best compliance (data physically separate), no noisy neighbors, tenant-specific optimization • Cons: Highest infrastructure cost, complex migrations, harder analytics across tenants • Best for: Enterprise SaaS, regulated industries (healthcare, finance), large tenants • Example: Salesforce orgs, some AWS RDS multi-tenant setups
4. Hybrid Approach
Different architectures for different tenant tiers. • Free/SMB tenants: Shared schema (cost-efficient) • Pro/Mid-market: Separate schema (better isolation) • Enterprise: Separate database (compliance + performance) • Migration path: Tenants can move up tiers as they grow • Pros: Optimized cost/performance for each segment • Cons: Most complex to implement and maintain • Best for: SaaS with clear tenant tiers (free/pro/enterprise)

Core Tables Every SaaS Needs

1. Tenants/Organizations:

CREATE TABLE tenants ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), name VARCHAR(255) NOT NULL, slug VARCHAR(255) UNIQUE NOT NULL, plan VARCHAR(50) DEFAULT 'free', -- free, pro, enterprise status VARCHAR(50) DEFAULT 'active', -- active, suspended, deleted settings JSONB DEFAULT '{}', created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW() );

2. Users:

CREATE TABLE users ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE, email VARCHAR(255) NOT NULL, password_hash VARCHAR(255), first_name VARCHAR(255), last_name VARCHAR(255), role VARCHAR(50) DEFAULT 'member', -- owner, admin, member, viewer status VARCHAR(50) DEFAULT 'active', last_login_at TIMESTAMP, created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW(), UNIQUE(tenant_id, email) );

3. Audit Log:

CREATE TABLE audit_logs ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE, user_id UUID REFERENCES users(id) ON DELETE SET NULL, action VARCHAR(100) NOT NULL, -- created, updated, deleted, exported resource_type VARCHAR(100) NOT NULL, -- customer, invoice, settings resource_id UUID, changes JSONB, -- { "before": {}, "after": {} } ip_address INET, user_agent TEXT, created_at TIMESTAMP DEFAULT NOW() );

4. Subscriptions/Billing:

CREATE TABLE subscriptions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE, plan VARCHAR(50) NOT NULL, status VARCHAR(50) DEFAULT 'active', -- active, canceled, past_due billing_cycle VARCHAR(50) DEFAULT 'monthly', -- monthly, annual price_cents INTEGER NOT NULL, currency VARCHAR(3) DEFAULT 'USD', current_period_start TIMESTAMP, current_period_end TIMESTAMP, cancel_at_period_end BOOLEAN DEFAULT FALSE, payment_method_id UUID, created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW() );

Indexing Strategies

1. Tenant isolation:

• Index every tenant_id column: CREATE INDEX idx_users_tenant ON users(tenant_id); • Composite indexes for common queries: CREATE INDEX idx_orders_tenant_status ON orders(tenant_id, status); • Partial indexes for filtered queries: CREATE INDEX idx_active_users ON users(tenant_id) WHERE status = 'active'; • Covering indexes: Include frequently queried columns to avoid table lookups

2. Query optimization:

• EXPLAIN ANALYZE: Run on every slow query before adding indexes • Index selectivity: Index columns with high cardinality (many unique values) • Write impact: Every index slows down writes; don't over-index • Index maintenance: Reindex periodically, especially after bulk loads

3. Partitioning:

• Time-based: Partition large tables by created_at (monthly, quarterly) • Tenant-based: Partition by tenant_id for very large tenants • Benefits: Faster queries, easier archival, parallel vacuum • Trade-offs: More complex queries, harder schema changes

Scaling Strategies

1. Read replicas:

• Separate read and write traffic • Route analytics, exports, and heavy queries to replicas • Async replication lag: 100ms-1s typical; handle stale reads • Automated failover: Promote replica if primary fails

2. Connection pooling:

• PgBouncer for PostgreSQL, ProxySQL for MySQL • Prevent connection exhaustion under load • Transaction vs. session pooling modes • Monitor: active connections, queue depth, wait time

3. Sharding:

• Horizontal partitioning: Split data across multiple databases • Shard key: Usually tenant_id for SaaS (all tenant data on one shard) • Rebalancing: Move tenants between shards as they grow • Cross-shard queries: Avoid if possible; use aggregation service if necessary

4. Caching:

• Redis/Memcached: Cache frequently accessed data (tenant settings, user permissions) • Query cache: Cache expensive query results (PostgreSQL has this built-in) • Application cache: Cache computed values (dashboard metrics, reports) • Cache invalidation: TTL-based, event-based, or manual

Database Design Checklist

☐ Multi-tenant architecture chosen (shared schema, separate schema, or separate DB) ☐ Tenant isolation implemented (tenant_id on every table) ☐ Core tables created (tenants, users, audit_logs, subscriptions) ☐ Indexes on all foreign keys and tenant_id columns ☐ Composite indexes for common query patterns ☐ Partial indexes for filtered queries ☐ EXPLAIN ANALYZE run on slow queries ☐ Soft deletes implemented (deleted_at instead of hard delete) ☐ Timestamps on all tables (created_at, updated_at) ☐ Audit logging configured for sensitive operations ☐ UUID primary keys (not auto-increment for security) ☐ JSONB columns for flexible schemas (settings, metadata) ☐ Constraints defined (NOT NULL, UNIQUE, CHECK, FOREIGN KEY) ☐ Connection pooling configured ☐ Read replicas set up (if needed) ☐ Backup strategy defined (point-in-time recovery, retention policy) ☐ Encryption at rest enabled ☐ Encryption in transit (TLS) enforced ☐ Database monitoring configured (slow queries, connection count, disk usage) ☐ Partitioning strategy defined (if tables >10M rows) ☐ Archival policy defined (move old data to cold storage) ☐ GDPR/data deletion process documented ☐ Migration strategy (Alembic, Flyway, or manual with rollback plans) ☐ Seeding strategy for new tenants (default data, templates) ☐ Load testing performed (simulate N concurrent tenants) ☐ Disaster recovery plan tested (restore from backup) ☐ Documentation: ERD, schema notes, query patterns
