🔌 API Documentation

Docs that developers love. Adoption that scales.

Why API Documentation is Critical

For developer-facing SaaS products, API documentation is the product. Developers don't evaluate your sales deck — they evaluate your docs. If they can't understand your API in 5 minutes, they won't integrate. If integration takes a week instead of a day, they'll choose your competitor.

Great API documentation:

• Reduces integration time from days to hours • Reduces support tickets from developers by 60-80% • Increases API adoption and usage • Enables self-service onboarding (no sales required) • Improves developer satisfaction and advocacy • Generates organic traffic (developers search for "how to do X with [API]")

API Documentation Elements

1. Getting Started
The first 5 minutes. Account creation, API key generation, first request. • Sign up for an account • Generate an API key (with security warnings) • Make your first API call (copy-paste ready) • Expected response format • Common errors and how to fix them Include a "Try it" button that works with test keys.
2. Authentication
How to authenticate requests. Be explicit. • API key in header (Authorization: Bearer sk_live_...) • OAuth 2.0 flow (if applicable) • Token expiration and refresh • Rate limits per key • Revocation process Include code examples in curl, Python, JavaScript, and Ruby.
3. Core Resources / Endpoints
Organized by resource, not by endpoint name. • /customers — Create, read, update, delete • /invoices — List, create, void, send • /webhooks — Subscribe, list, delete For each endpoint: - HTTP method and URL - Request parameters (required vs. optional, types, defaults) - Request body schema (JSON example) - Response schema (success and error) - Status codes and their meanings - Code examples in 3+ languages
4. Error Handling
Every error code explained with solutions. • 400 Bad Request — Invalid parameters (check types and required fields) • 401 Unauthorized — Invalid API key (generate a new one) • 403 Forbidden — Insufficient permissions (upgrade plan or check scopes) • 404 Not Found — Resource doesn't exist (check ID) • 429 Too Many Requests — Rate limit hit (exponential backoff) • 500 Internal Server Error — Retry with exponential backoff, contact support if persistent Include error response format: { "error": { "code": "...", "message": "...", "details": "..." } }
5. Pagination
How to handle large result sets. • cursor-based (preferred for real-time data) • offset/limit (simpler but less reliable for live data) • Include total count, has_more flag, and next cursor • Code example showing how to iterate through all results
6. Webhooks
Event-driven integration. • How to subscribe to webhooks • Event types and payload formats • Retry logic (exponential backoff) • Signature verification (security) • Idempotency (prevent duplicate processing) • Testing webhooks locally (ngrok, Stripe CLI)
7. SDKs and Libraries
Official wrappers make integration easier. • npm install your-sdk • pip install your-sdk • gem install your-sdk • composer require your/sdk Maintain official SDKs for top 3 languages. Community SDKs for the rest. SDK docs should mirror API docs but with language idioms.
8. Changelog & Versioning
API changes break integrations. Communicate clearly. • Semantic versioning for API (v1, v2) • Deprecation policy (6-12 months notice) • Migration guides between versions • Sunsetting dates for deprecated endpoints • Changelog with all additions, changes, and removals Email developers about breaking changes 90 days in advance.

Documentation Best Practices

• Copy-paste ready: Every code example should work if pasted into a terminal or IDE. No placeholders like without instructions.

• Multiple languages: At minimum: curl, JavaScript, Python. Add Ruby, PHP, Go, Java based on your audience.

• Interactive console: Let developers make real API calls from the docs. Stripe's "Try it" feature is the gold standard.

• Real responses: Show actual JSON responses, not simplified versions. Developers need to see edge cases, null values, and nested structures.

• Search: Developers search for endpoints, error codes, and concepts. Algolia DocSearch is the standard.

• Mobile-friendly: Developers read docs on phones during commutes, on tablets while testing. Responsive design is essential.

• Dark mode: Developers expect it. It's not a nice-to-have anymore.

API Documentation Tools

All-in-one platforms:

• ReadMe: The gold standard. Interactive API console, changelog, community. Paid but worth it. • Stoplight: Design-first API docs. Visual OpenAPI editor + documentation. • Bump.sh: Auto-generated from OpenAPI. Clean, modern design. • Postman Public Workspaces: Collections + documentation + collaboration. Free tier available.

Open source / Static site:

• Swagger UI: Classic OpenAPI renderer. Functional but dated design. • Redoc: Beautiful OpenAPI documentation. Three-panel layout. Highly customizable. • Docusaurus + OpenAPI: Full documentation site with API reference. Used by Meta, Supabase. • Slate: Static API docs. Clean three-column layout. Used by Stripe (original version). • GitBook: Good for mixed content (guides + API reference). Less interactive than ReadMe.

Auto-generation:

• OpenAPI Generator: Generate docs, SDKs, and server stubs from OpenAPI spec. • Swagger Codegen: Similar to OpenAPI Generator. • Postman: Convert collections to documentation. • Insomnia: Generate docs from request collections.

OpenAPI Specification

OpenAPI (formerly Swagger) is the industry standard for API description. A single YAML/JSON file describes your entire API:

• Benefits: Single source of truth. Generate docs, SDKs, tests, and mock servers from one file. • Version: Use OpenAPI 3.0 or 3.1. 2.0 is legacy. • Validation: Validate requests/responses against spec in CI/CD. Catch breaking changes. • Code-first vs. Design-first: Code-first generates spec from code annotations. Design-first writes spec, then generates code. Design-first is preferred for public APIs. • Tools: Swagger Editor (online), Stoplight Studio (desktop), Insomnia (design + test), Postman (import OpenAPI).

Developer Experience (DX)

API documentation is part of a broader developer experience:

• Onboarding: Sign up → API key → first request in < 5 minutes. Stripe does this perfectly. • Sandbox: Test environment with fake data. No risk, no real charges. • Postman collection: Import and start testing immediately. Pre-filled with examples. • Status page: API uptime, latency, incident history. Builds trust. • Community: Forum, Discord, Stack Overflow tag. Developers help each other. • Support: Developer-focused support (not generic). Fast response to API questions. • Examples: Real-world use cases, not just endpoint reference. "Build a subscription system" tutorial.
