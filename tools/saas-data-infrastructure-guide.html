SaaS Data Stack

SaaS products generate massive amounts of data: user events, transactions, analytics, logs, and more. Your data infrastructure must handle growth without requiring constant rewrites.

The modern SaaS data stack:

📊 Application Database
PostgreSQL for relational data. Flexible, reliable, supports JSON. Good for 99% of SaaS apps. When you outgrow single-node Postgres, add read replicas or shard.
📈 Analytics/Event Store
ClickHouse, BigQuery, or Snowflake for high-volume events. Separate from application DB to avoid slowing down user-facing queries. Events → Kafka/Redpanda → Data warehouse.
⚡ Cache Layer
Redis for hot data: sessions, rate limits, leaderboards, real-time features. Reduces DB load by 80-90% for read-heavy workloads.
🔍 Search
Elasticsearch or Meilisearch for full-text search. Don't query Postgres with LIKE '%term%' — it's slow and doesn't scale. Sync data via change data capture (CDC).
📁 Object Storage
S3, R2, or MinIO for files, images, exports. Cheap, infinite scale, CDN-compatible. Never store user uploads on local disk.
📡 Message Queue
Redis, RabbitMQ, or Kafka for async jobs: emails, webhooks, exports, reports. Decouples heavy work from user requests. Prevents timeouts and improves UX.

Database Scaling Strategy

Phase 1 (0-1K users): Single-node Postgres. One app server. One Redis instance. Keep it simple.

Phase 2 (1K-10K users): Add read replicas for reporting. Add Redis for caching. Move long jobs to queues. Add CDN for static assets.

Phase 3 (10K-100K users): Shard by tenant (if multi-tenant) or by user ID. Add connection pooling (PgBouncer). Move analytics to separate data warehouse. Implement CDC for search sync.

Phase 4 (100K+ users): Consider specialized databases per workload. CockroachDB or Spanner for global distribution. Kafka for event streaming. Data lake for ML training data.

Rule: Don't optimize for Phase 4 when you're at Phase 1. But architect so Phase 2 is easy. Multi-tenancy, connection pooling, and queue-based jobs from day one.

Data Architecture Patterns

1. Event Sourcing: Store events, not state. "User signed up," "User upgraded," "User churned." Reconstruct any state by replaying events. Audit-friendly, scalable, complex.

2. CQRS (Command Query Responsibility Segregation): Separate write and read models. Writes go to transactional DB. Reads go to optimized views (Redis, Elasticsearch, materialized views). Complex but powerful for read-heavy apps.

3. Multi-Tenancy: Shared database, shared schema with tenant_id column. Or shared database, separate schema per tenant. Or separate database per tenant. Tradeoffs: cost vs. isolation vs. complexity.

4. Change Data Capture (CDC): Stream database changes to other systems. Postgres → Debezium → Kafka → Elasticsearch/warehouse. Real-time sync without polling.

Data Security & Compliance

Encryption: At rest (AES-256) and in transit (TLS 1.3). Non-negotiable. Use managed keys (AWS KMS, Google Cloud KMS) for key rotation.

Access control: Role-based access control (RBAC) for internal tools. Audit logs for all data access. Principle of least privilege: employees get minimum access needed.

GDPR/CCPA: Right to access, right to deletion, right to portability. Build these into your data model from day one. "Hard delete" vs. "soft delete." Data retention policies. Consent tracking.

SOC 2 / ISO 27001: Security policies, access reviews, incident response, penetration testing. Required for enterprise SaaS. Start documenting practices early — auditors love paper trails.

Data Infrastructure Metrics

• Query response time: P95 < 200ms for user-facing queries.
• Database CPU: < 70% sustained. Spike to 90% during batch jobs is okay.
• Cache hit rate: > 80% for Redis. Lower = missing opportunities.
• Queue depth: Near-zero most of the time. Growing queue = workers can't keep up.
• Replication lag: < 1 second for read replicas. Higher = stale data.
• Storage growth: Track month-over-month. Plan capacity 3 months ahead.
