What are Webhooks?

Webhooks are user-defined HTTP callbacks. Instead of your app polling our API every minute ("Did anything change?"), we push events to your URL in real-time ("Something changed!"). They're the backbone of modern SaaS integrations.

Polling: You ask us every minute. 1,440 requests/day. 99% return nothing. Wasteful, slow, expensive.
Webhooks: We tell you when something happens. Instant. Efficient. Scalable.

Every major SaaS uses webhooks: Stripe (payment events), Slack (message events), GitHub (push events), Twilio (SMS delivery). If you're building a SaaS, you need webhooks.

Webhook Design Principles

🎯 Event-Driven Architecture
Webhooks are notifications, not commands. "Payment succeeded" not "Charge this card." Keep webhooks stateless and idempotent. The receiver decides what to do with the event.
🔐 Signature Verification
Sign every payload with HMAC-SHA256. Include signature in header (e.g., X-Webhook-Signature). Receivers verify before processing. Prevents tampering and replay attacks. Never send secrets in the payload.
🔄 Idempotency
Same event ID = same outcome. Receivers should handle duplicate webhooks gracefully (network retries, redeliveries). Include event_id in payload. Store processed IDs for 24+ hours.
⏱️ Retry with Backoff
If receiver returns non-2xx, retry with exponential backoff. 1 min, 2 min, 4 min, 8 min... up to 24 hours. Max 3-5 retries for most events, 10+ for critical (payments, security). Don't spam failed endpoints.
📋 Event Payload Design
Include event metadata (id, type, timestamp) and resource snapshot (current state, not just delta). "invoice.paid" should include the full invoice object, not just "invoice_id": "inv_123". Reduces API calls for receivers.
🧪 Test Endpoints
Provide a way to test webhooks without real events. Dashboard "Send Test Webhook" button. CLI tool. Postman collection. Developers need to verify their handler before going live.

Webhook Event Types

Name events consistently: resource.action

• Lifecycle: customer.created, customer.updated, customer.deleted
• Billing: invoice.created, invoice.paid, invoice.payment_failed, subscription.canceled
• Usage: report.generated, export.completed, limit.approaching
• Security: user.login, api_key.created, permission.changed, mfa.enabled
• Integration: integration.connected, integration.disconnected, sync.completed

Document every event: trigger condition, payload schema, example payload, retry policy. Maintain a changelog when events are added, modified, or deprecated.

Implementing Webhooks (Sender)

1. Registration: User provides URL + optional secret + event types they want. Validate URL (HEAD request, check 2xx). Store with webhook_id.

2. Delivery: Queue event → worker picks up → POST to URL with payload + headers. Timeout: 5-30 seconds. Don't block user requests.

3. Signature: Sign payload with shared secret: HMAC-SHA256(secret, timestamp + "." + payload). Include timestamp to prevent replays. Receiver verifies: extract timestamp, reconstruct signature, compare.

4. Retries: On failure, requeue with delay. Exponential backoff: 1 min → 2 min → 4 min → 8 min → 15 min → 30 min → 1 hour → 2 hours → 4 hours. Max attempts: 5-10. After max: mark as failed, notify user.

5. Monitoring: Track delivery rate, latency, failure rate per endpoint. Alert users if their endpoint has been failing for > 24 hours. Provide delivery logs in dashboard.

Implementing Webhooks (Receiver)

1. Verify signature: Before processing anything. Reject if invalid. Use constant-time comparison to prevent timing attacks.

2. Check timestamp: Reject if timestamp is > 5 minutes old. Prevents replay attacks.

3. Acknowledge quickly: Return 200 immediately, then process async. Webhooks should be fast. If processing takes > 30 seconds, the sender may retry (duplicate processing).

4. Handle duplicates: Store processed event IDs. Check before processing. Return 200 for duplicates (sender should stop retrying).

5. Queue for processing: Don't process webhook payload directly in the HTTP handler. Push to internal queue (Redis, RabbitMQ, SQS). Worker processes asynchronously. Prevents timeouts and handles bursts.

6. Graceful degradation: If your service is down, sender retries. When you come back, you'll catch up. Design for eventual consistency.

Webhook Dashboard Features

• Endpoint management: Add, edit, delete webhook URLs. Test connectivity.
• Event filtering: Subscribe to specific event types. Reduce noise.
• Delivery logs: History of sent webhooks with status, response code, latency, payload.
• Retry controls: Manual retry for failed deliveries. Pause/resume endpoints.
• Secret rotation: Generate new secrets without downtime. Support multiple active secrets during transition.

Webhook Metrics

• Delivery rate: % of webhooks delivered successfully. Target: > 99.5%.
• Latency (P95): Time from event to delivery. Target: < 5 seconds.
• Failure rate by endpoint: Identify problematic receiver URLs.
• Retry rate: % requiring retries. High = unreliable receivers or network issues.
• Endpoint health: % of endpoints failing for > 24 hours. Alert users to fix.
