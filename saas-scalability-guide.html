📈 Scalability

Handle 10x growth. Never rebuild from scratch.

Scalability Principles

Scalability is the ability of your system to handle growth without proportional increases in cost or decreases in performance. The best time to design for scale is before you need it. Retrofitting scalability is 10x harder than building it in from the start.

Key principle: Stateless application servers, horizontal scaling, caching layers, and async processing. These four patterns handle 90% of SaaS scaling needs.

Scaling Strategies

↔️ Horizontal Scaling (Scale Out)
Add more servers instead of bigger servers. Stateless app servers behind a load balancer. If one server handles 1,000 users, 10 servers handle 10,000. Cheaper, more resilient, no single point of failure. Requires stateless architecture and session externalization (Redis).
⚡ Caching
Store frequently accessed data in fast storage (Redis, Memcached, CDN). Database queries: 5-50ms. Cache hits: 0.5ms. Cache 80% of reads and reduce DB load by 80%. Cache at every layer: browser, CDN, application, database query cache.
📨 Async Processing
Move slow work out of the request path. Email sending, report generation, image processing, webhooks — all go to a queue. User gets instant response ("Report queued"), worker processes in background. Prevents timeouts and improves perceived performance.
🗄️ Database Optimization
Indexes on query columns, query optimization, read replicas for reporting, connection pooling (PgBouncer), materialized views for complex aggregations. The database is always the bottleneck — treat it with respect.
📊 Read Replicas
Write to primary database, read from replicas. Most SaaS apps are read-heavy (80% reads, 20% writes). Read replicas handle reporting, analytics, and heavy queries without impacting primary. Keep replication lag < 1 second.

Database Scaling Phases

Phase 1 (0-1K users): Single-node database. No optimization needed. Focus on product-market fit.

Phase 2 (1K-10K users): Add indexes on frequently queried columns. Add Redis for caching and sessions. Add read replicas for reporting. Monitor slow queries with pg_stat_statements or equivalent.

Phase 3 (10K-100K users): Connection pooling (PgBouncer). Shard by tenant or time range. Move analytics to separate data warehouse (BigQuery, Snowflake). Implement CDC for search sync.

Phase 4 (100K+ users): Consider specialized databases per workload (OLTP, OLAP, search, cache). Distributed databases (CockroachDB, Spanner) for global distribution. Event streaming (Kafka) for real-time data.

Load Testing

You can't optimize what you don't measure. Load testing reveals bottlenecks before customers do:

Tools: k6, Artillery, Locust, JMeter, Gatling. k6 is developer-friendly (JavaScript-based) and integrates with CI/CD.

Scenarios:

• Baseline: 100 concurrent users, normal usage pattern
• Spike: 10x traffic for 5 minutes (viral moment, product launch)
• Endurance: 8 hours at 2x normal load (memory leaks, connection pool exhaustion)
• Stress: Gradually increase until system breaks (find the ceiling)

Metrics: Response time P95/P99, error rate, throughput (req/s), resource utilization. Compare against SLOs. If P95 latency > 500ms at target load, you have work to do.

Scalability Anti-Patterns

❌ N+1 Queries: Loading a list of 100 items, then querying the DB 100 times for related data. Fix with eager loading (JOINs) or caching.

❌ Giant Monolith: One codebase doing everything. Hard to scale pieces independently. Consider microservices only when you have 50+ engineers and clear service boundaries.

❌ No Caching: Querying the database for data that rarely changes. Redis is cheap. Database queries are expensive.

❌ Synchronous Everything: Waiting for email to send before returning HTTP response. Move slow operations to queues.

❌ Over-Engineering: Microservices, Kubernetes, and event sourcing for a product with 100 users. Start simple. Add complexity when metrics justify it.

Capacity Planning

Plan capacity 3-6 months ahead:

1. Measure current: CPU, memory, disk, network, database connections, cache hit rate, queue depth.

2. Project growth: Based on user growth, data growth, and feature launches. If you're growing 20% MoM, you'll 10x in ~12 months.

3. Identify bottlenecks: What's at 70%+ utilization? That's your next wall. Fix it before you hit it.

4. Plan scaling: When do you need more servers? More database storage? More cache memory? Budget and procure ahead of time.

5. Test projections: Run load tests at projected future load. Does the system handle it? If not, fix before you need it.

Scalability Metrics

• Requests per second: Total throughput. Growing = good.
• Response time (P95/P99): Latency at the tail. P95 < 200ms, P99 < 500ms.
• Error rate: % of 5xx errors. < 0.1% is healthy.
• Resource utilization: CPU, memory, disk. < 70% sustained = headroom.
• Cache hit rate: % of requests served from cache. > 80% is good.
• Queue depth: Number of pending async jobs. Near-zero = healthy.
• Cost per user: Infrastructure cost divided by user count. Should decrease with scale (economies of scale).
